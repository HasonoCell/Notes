## 一、背景
2026 年 2 月，OpenAI 发布了一篇看似普通的[技术博客](https://openai.com/zh-Hans-CN/index/harness-engineering/)，却在工程师圈子里引爆了一场关于"软件工程未来"的深刻争论。

他们描述了这样一件事：**3 名工程师，用了 5 个月，几乎不写任何代码，却构建出了一个超过百万行的真实产品。**他们用自家的 Codex AI Agent，5 个月内，3 人合并了约 1500 个 Pull Request，平均每人每天 3.5 个 PR。最终产物包含应用逻辑、基础设施、工具链、文档和内部开发者工具，团队估算整体效率比人工编码快了约 10 倍。

Thoughtworks 工程师 Birgitta Böckeler 随后发表了[深度分析](https://martinfowler.com/articles/exploring-gen-ai/harness-engineering.html)，被软件工程师 Martin Fowler 誉为"对 AI 辅助软件开发核心部分的有价值框架"。她把这套理念正式命名为 **Harness Engineering**。

## 二、什么是 Harness Engineering？
### "Harness" 这个词从哪来？
"Harness"是英文里**马具**的意思——缰绳、马鞍、嚼子——是引导一匹强壮但不可预测的马朝正确方向前进的完整装备体系。

这个比喻是有意为之的，非常精准：

|比喻元素|对应含义|
|-|-|
|**马**|AI 模型——强大、快速，但本身不知道该往哪里走|
|**马具**（缰绳、马鞍、嚼子）|工程师构建的那套系统：文档、约束、工具、反馈循环|
|**驾马人**|工程师——不再亲自奔跑，而是掌控方向|

没有马具，马跑得很快，但方向混乱、不可控；有了马具，同一匹马就能完成精准的工作。

### 定义
**Harness Engineering** 是以"驾驭 AI"为核心职责的新一代软件工程范式——人类工程师不再亲自编写代码，而是专注于构建让 AI Agent 能够可靠、高质量地完成工程任务的整套"驾驭系统"。

它本质上是一门关于 **"如何与 AI 协同完成大规模工程"** 的学科，回答三个核心问题：

1. **如何让 AI 知道该做什么？** → 上下文工程（文档、地图、意图描述）
2. **如何让 AI 不做错误的事？** → 架构约束（Linter、结构测试、边界规则）
3. **如何让 AI 持续做对的事？** → 反馈循环 + 熵管理（Agent 互审、定期清理、质量监控）

### 传统工程 vs Harness Engineering
|维度|传统软件工程|Harness Engineering|
|-|-|-|
|工程师做什么|写代码|构建让 AI 写好代码的环境|
|核心技能|编程语言、算法、调试|上下文设计、约束建模、反馈系统|
|代码从哪来|人类手写|AI Agent 生成|
|质量如何保证|Code Review、单元测试|架构约束 + Agent 互审 + 熵管理|
|工程师关注层级|实现细节|意图、结构、方向|



## 三、Harness 的四大支柱
Harness Engineering 不是一个单一工具，而是由四个相互依存的支柱共同构成的工程体系。



### 支柱一：上下文工程（Context Engineering）
> **核心原则：给 Agent 一张地图，而不是一本千页手册。**
上下文（Context）是 AI Agent 的"工作记忆"。它同时包含：任务描述、代码库结构、约束规则、历史对话……这些信息共享同一个有限的"上下文窗口"。

OpenAI 团队最初尝试了"一个巨大的 AGENTS.md 文件"方案——把所有规则、背景和约束都塞进一个文档。结果以可预见的方式失败了：这个文件太大，会挤占任务本身和相关代码的空间，导致 Agent 要么遗漏关键约束，要么开始优化错误的目标。

**最终方案：分层文档体系**

* 一个约 **100 行**的简短 `AGENTS.md` 作为"导航地图"
* 指向 `docs/` 目录下更深层的设计规范、执行计划、领域约定
* Agent 按需获取信息，而不是一次性加载全部

LangChain CEO Harrison Chase 进一步总结了上下文工程的趋势：**把控制权交给 LLM 本身**——让模型自己决定在某个时刻需要看什么、不看什么，在合适时机主动压缩上下文。

核心原则是：**在正确的时间、以正确的格式，把正确的信息送达 LLM**。



### 支柱二：架构约束（Architectural Constraints）
> **核心原则：让规则成为 Agent 的反射动作，而不是事后审查。**
OpenAI 为代码库执行了严格的分层架构：

* 每个业务域被划分为固定层级
* 依赖方向严格单向（如：业务层不能调用基础设施层的内部实现）
* 只允许有限且明确的跨层边界

这些约束通过**自定义 Linter**（静态代码分析工具）和**结构性测试**来机械地执行——而这些 Linter 本身也是由 Codex 生成的。

在人类主导的工作流中，这些规则可能显得过于严苛；但在 Agent 场景下，它们变成了**倍增器**：规则一旦被编码，就在所有地方同时生效，无需人工监督每一行代码。

**最聪明的设计细节**：自定义 Linter 的错误信息本身就是修复指引。当 Agent 违反架构约束时，错误信息不只是标记违规，还告诉 Agent **如何修复**——工具在工作的同时，教会 Agent 正确的做法。



### 支柱三：闭环反馈（Recursive Feedback Loop）
> **核心原则：让 Agent 之间互相挑剔，把人类的注意力留给真正重要的决策。**
OpenAI 将他们的反馈循环称为 **"Ralph Wiggum 循环"**（来自《辛普森一家》里那个天真烂漫的小孩，暗指循环的自动化和天真性）：

```
Agent 执行任务 → 开一个 PR
  → 自我审查本地变更
  → 向其他 Codex 实例请求特定角度的代码审查
  → 响应反馈、修改代码
  → 迭代，直到所有审查者满意
  → 提交给人类工程师做最终确认
```
大多数"挑剔"发生在 **Agent 与 Agent 之间**，让人类工程师保持在高层抽象层面，专注于产品方向而非语法细节。

**可观测性基础设施**的接入是另一个关键。Agent 被连接到日志、指标、链路追踪系统，以监控应用性能并在隔离环境中复现 Bug。OpenAI 甚至将 **Chrome DevTools Protocol** 接入 Agent 的运行时，让 Agent 可以截图、操作 DOM、自主导航应用界面，从而独立修复视觉 Bug。



### 支柱四：熵管理（Entropy Management）
> **核心原则：技术债要每日偿还，而不是让它积累成雪崩。**
完全的 Agent 自治会引入**熵**（Entropy）——系统的无序度。

AI Agent 天然倾向于**复制现有模式**，哪怕是不好的模式。随着时间推移，代码库会悄悄漂离最初的架构设计，产生越来越多的"聪明但有害"的代码。

OpenAI 最初的应对方式是每周拿出 20% 的时间（通常是周五全天）清理"AI 产出的糟粕"。这显然不可扩展。

**最终解决方案**：将黄金原则直接编码进仓库，并构建**自动化的持续清理流程**：

* 后台 Codex 任务定期扫描代码库，检测偏离"黄金标准"的地方
* 自动更新质量评分，开出针对性的重构 PR
* 大多数清理 PR 在**一分钟内**被审查和自动合并

技术债被当作"高利贷"每日偿还，而不是积累到无力回填。



## 四、工程师角色的根本转变
Martin Fowler 指出了 OpenAI 那篇文章里最深刻的一点：**"Harness" 这个词在正文里只出现了一次，但它才是整篇文章真正的标题。**

工具链、实践方式、让 Agent 生成的代码在规模上可维护的约束体系——这才是一切的核心。



在传统工作流中，工程师花大量时间将需求翻译成语法——变量命名、函数结构、循环逻辑……这些都需要人脑一行行思考。

在 Agent 优先的工作流中，**语法已成商品**。任何 AI Agent 都能生成符合语法的代码；真正稀缺的是：正确的意图、清晰的结构、可靠的方向。

OpenAI 团队的主要职责因此转向了三件事：

1. **设计环境**：构建 Agent 能够高效工作的上下文、工具和约束
2. **明确意图**：把产品需求转化为 Agent 能理解的结构化描述
3. **构建反馈循环**：让 Agent 的产出持续对齐人类的质量预期

### 
若 Agent 产生 10 倍的代码量，人工审查者将立刻成为最严重的瓶颈。

Harness Engineering 的解法是**分层审查**：

* **Agent 对 Agent**：处理机械化检查——格式、规范、架构约束违规
* **人类工程师**：专注于真正需要判断力的决策——新颖的架构方案、安全敏感变更、产品方向

这不是降低了工程师的地位，而是将其提升到了更高的抽象层——从"打字员"变成了"系统设计者"。



## 五、争论：Harness 会被更强的模型取代吗？
这是当前业界最深刻的分歧，值得认真对待。

### 观点一："模型是商品，Harness 是护城河"
**数据证明 Harness 的价值是真实的。**

2026 年 3 月，研究者 Nate B Jones 做了一个对照实验：同一个底层 AI 模型，仅仅因为 Harness 不同，在编码 benchmark 上的成功率可以从 **42% 跳升至 78%**——翻了近一倍，而模型本身分毫未变。

LangChain 的案例更加直接：他们在 Terminal Bench 2.0 排行榜上从 **Top 30 跃升至 Top 5**，编码 Agent 成功率从 52.8% 升至 66.5%，对底层模型没有做任何改变，只改变了 Harness 的设计。

还有更多佐证：

* **Manus** 在 Harness 上重写了四次，每次都带来显著的性能提升
* **LangChain** 花了一年时间经历四种架构迭代
* **Vercel** 从"提供所有工具"退回到"只保留核心工具"——Agent 反而更快、更可靠

这些 Harness 优势**无法从 Hugging Face 下载**，必须自己构建、测试、失败、重来。先建好 Harness 的公司拥有结构性的竞争优势。

### 观点二：更强的模型终将使 Harness 过时
强化学习先驱 Rich Sutton 的 **"苦涩的教训"（Bitter Lesson）** 指出：依赖计算的通用方法，终将击败人类手工编码的知识。

这个逻辑延伸到 Harness 领域，意味着：足够强大的模型最终可以在不需要大量 Harness 的情况下，端到端地完成复杂任务。

Manus 不得不随着模型进化重建四次 Harness，本身就是"模型进步侵蚀 Harness 价值"的有力证明。

但 Harness 派的核心反驳是：

> **多步骤执行任务存在不可简化的协调需求。**
上下文管理、状态持久化、错误恢复——这些不是"让模型更聪明"就能解决的推理问题，而是**需要系统来处理的基础设施问题**。一个模型不需要"更聪明"才能把进度保存到磁盘；它需要的是一个持久化状态的 Harness。



### 两派观点对比
|维度|Harness 派|模型能力派|
|-|-|-|
|核心主张|模型是引擎，Harness 是赛车|足够强的模型可以自主导航|
|关键证据|LangChain 同模型提升 26%；Nate B Jones 42%→78%|Manus 四次重写 Harness 随模型进化|
|实践建议|投资上下文、架构约束、反馈循环|保持 Harness 轻量、可撕裂|
|长期预测|Harness 成为组织护城河|更强模型将吸收部分 Harness 职责|



### 共识：Harness 必须是轻量、可撕裂的
这场争论催生了一个实践共识：**Harness 必须是轻量的（Lightweight）、可撕裂的（Rippable）。**

每次新模型发布，都有不同的最优 Agent 结构方式：

* 2024 年需要复杂手工管道才能处理的能力，在 2026 年已被单个 Prompt 处理
* 如果过度工程化控制流，下一次模型更新就会打破你的系统
* 开发者必须构建允许自己"撕掉昨天的智能逻辑"的 Harness



LangChain CEO Harrison Chase 给出了一个务实的判断：**随着模型变得更智能、更有能力，围绕它们的 Harness 也必须同步进化。** Harness 的演化趋势是把更多控制权交给 LLM 本身，让它自主决定如何管理上下文，而不是人工硬编码一切。

更长远的视角：我们正朝着**训练环境与推理环境的融合**前进。Harness 将成为解决"模型漂移"的主要工具——各大实验室会利用 Harness 检测模型何时在执行到第 100 步之后开始偏离指令，这些数据将被反馈到训练中，创造出在长任务中不"疲惫"的模型。



## 六、对软件工程师的实践启示
如果你是一名工程师，Harness Engineering 对你意味着什么？

**短期（现在就能做的事）**：

* 在你的 AI 辅助开发中，用分层文档代替单一的巨型提示词
* 为 AI 生成的代码建立自动化 Linter，并让错误信息包含修复指引
* 用代码审查的时间建立架构约束，而不是逐行审查 AI 输出

**中期（团队层面的投资）**：

* 构建 Agent 互审流程，把人类从机械化检查中解放出来
* 建立代码质量的自动化监控，让技术债每日可见
* 将架构规则编码进测试套件，而不只是写进文档

**长期（职业方向的思考）**：

* 你的核心价值将从"写出正确代码"转向"构建能产出正确代码的系统"
* 上下文设计、约束建模、反馈系统将成为新的稀缺技能
* 先掌握 Harness 思维的工程师，将在 AI 时代拥有结构性优势
