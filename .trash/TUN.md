**TUN 和 TAP 是 Linux 内核中的虚拟网络设备。**

要理解它们，我们先要对比一下真实的物理网卡（比如 `eth0` 或你的 Wi-Fi 网卡 `wlan0`）是怎么工作的：

- **物理网卡：** 一端连着**真实的物理网线**（负责接收光电信号），另一端连着 **Linux 内核的网络协议栈**。
    
- **TUN / TAP 网卡：** 一端连着 **Linux 内核的网络协议栈**，另一端连着的不是网线，而是一个**用户态的应用程序**（比如你写的 Go 程序、VPN 软件、或者虚拟机）。
    

对 Linux 内核来说，TUN/TAP 设备和真正的物理网卡没有任何区别。内核依然会给它分配 IP 地址、设置子网掩码、配置路由表，并向它发送数据包。

真正的魔法在于它背后的实现：**它是一根连接“操作系统内核”与“用户空间程序”的水管。**

### 1. 它是怎么工作的？（万物皆文件）

当你在 Linux 中创建了一个 TUN 设备（比如叫 `tun0`）时，操作系统会在 `/dev/net/tun` 下暴露一个文件描述符（File Descriptor）给你的应用程序。

- **发送数据（应用 -> 内核 -> 外网）：** 当你的程序往这个文件里 `Write()` 一串字节流时，内核会以为这是从“网线”上接收到的真实数据包，然后把它推入内核的网络协议栈进行处理，最后可能从真实的物理网卡转发出去。
    
- **接收数据（内核 -> 应用）：** 当系统里的其他程序（比如浏览器）试图把数据发给 `tun0` 的 IP 时，内核会把这些网络包转成字节流。你的程序只需要对这个文件执行 `Read()` 操作，就能把网络包“拦截”到自己的进程内存里。
    

### 2. TUN 和 TAP 的核心区别：工作在第几层？

它们俩的原理一模一样，唯一的区别是它们处理的数据包的“层级”不同：

- **TUN (Network TUNnel - 网络隧道)：** 工作在**第三层（网络层 / IP 层）**。
    
    - 你从 TUN 读写的数据，直接就是 **IP 数据包**（包含源 IP 和目的 IP，没有 MAC 地址）。
        
    - **用途：** 因为它不关心底层的物理 MAC，只关心 IP 路由，所以它天生适合做 **VPN（虚拟专用网）** 或我们之前讨论的**用户态网络栈**。比如 WireGuard、OpenVPN 都是基于 TUN 实现的。
        
- **TAP (Network TAP - 网络监听)：** 工作在**第二层（数据链路层 / MAC 层）**。
    
    - 你从 TAP 读写的数据，是完整的**以太网帧（Ethernet Frame）**（既有 IP 头，也有外层的 MAC 地址头）。
        
    - **用途：** 因为它连 MAC 地址都能模拟，所以它看起来就像一根插在局域网交换机上的真实网线。它极其适合用来做 **虚拟机网络（Virtual Machines）**。比如当你用 KVM/QEMU 启动一台虚拟机时，宿主机上就会创建一个 TAP 设备，虚拟机的虚拟网卡发出的所有带有 MAC 地址的底层帧，都会通过这个 TAP 设备传递给宿主机的网络。
        

### 3. 最经典的实战场景：VPN 是怎么用 TUN 的？

我们可以用你平时用的“翻墙”软件或公司内网 VPN 来推演一下 TUN 的工作流：

1. VPN 软件启动，向内核申请创建了一个 `tun0` 虚拟网卡，并拿到了文件句柄。内核给 `tun0` 分配了 IP `10.8.0.2`。
    
2. VPN 软件修改了操作系统的**路由表**：强制规定“所有发往国外的 IP，统统扔给 `tun0` 网卡”。
    
3. 你的浏览器想访问 `google.com`。它构造了一个普通的 HTTP/IP 数据包。
    
4. 内核查路由表，把这个包扔进了 `tun0`。
    
5. 此时，包并没有发到网线上，而是**掉进了 VPN 软件的程序内存里**（因为 VPN 在不断 `Read` 这个文件）。
    
6. VPN 程序拿到这个原始包，对其进行**加密**，然后把它当作一段普通的业务数据，塞进一个全新的 UDP 数据包里，目标地址写上海外 VPN 服务器的公网 IP。
    
7. 这个全新的 UDP 包通过**真实的物理网卡 (`eth0`)** 发送到公网。
    

在这个过程中，你的浏览器以为自己把包发给了真正的网络，物理网卡以为自己只是在发普通的 UDP 流量，而 **TUN 设备就是欺骗这两者、并把数据“偷”到应用程序里进行二次加工的关键桥梁。**
