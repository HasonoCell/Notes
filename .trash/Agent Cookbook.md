Anthropic Agent Cookbook

# 背景

Anthropic有一系列的博客讲述他们对Agent系统搭建的一些经验，从24年1月到 现在（26年3月）都在更新，内容包括**Agent的基本模块、Agent架构类型以及如何选择Agent架构**等。对我们后面Agent架构的应用有很大的启发。

原文链接：[Building Effective AI Agents: Architecture Patterns and Implementation Frameworks](https://resources.anthropic.com/hubfs/Building%20Effective%20AI%20Agents-%20Architecture%20Patterns%20and%20Implementation%20Frameworks.pdf?hsLang=en)

这篇博客主要讲的是：**企业该如何真正把 AI Agent 做成可落地、可扩展、可治理的生产系统**，核心不是炫技，而是用合适的架构解决合适的问题。 

# 核心观点

文章首先提出了生成式AI和AI Agent的主要区别，用于后续区分两种形式的使用场景。

* **生成式AI更适合回答问题**
* **AI Agent更适合解决问题**

也就是说，Agent 不只是输出文本，而是能围绕目标自主拆解任务、调用工具、检查结果、持续迭代，直到完成任务或触发人工介入。文中强调，Agent 的价值尤其体现在那些**流程复杂、步骤不固定、需要动态决策**的场景，比如客服、数据分析、研发、法务、营销、风控等。

# Agent设计原则
### **从简单开始，逐步升级**

从简单的单一Agent做好一个任务开始搭建，逐步扩展成更复杂的系统。单一Agent可以更容易发现其中出现的错误，并且搭建的成本更低。

### **模型选型要匹配任务**

不是所有任务都要上最强模型。复杂推理、高价值决策适合强模型；简单抽取、批量处理适合更轻更快更便宜的模型。

### **模块化设计**

Prompt、Tools、Skills、Agent 角色最好都拆成可复用模块，方便后续替换模型、扩展能力、升级架构。

### **从一开始就建设可观测性**

Agent 系统不是传统程序，很多时候“为什么错了”比“错了没有”更难排查，所以要重点记录 **prompt 链路、工具调用、上下文、决策路径、token 消耗、记忆读写**等。



# 单Agent和多Agent架构介绍

单Agent是很多场景的应用起点，当单 Agent 顶不住时，就要考虑多 Agent。

### **单Agent架构**

工作循环：接收任务 → 规划 → 调工具执行 → 观察结果 → 调整策略 → 继续执行

**单Agent组成**：

* 大模型推理引擎
* System Prompt
* 一组可调用工具
* 一些可插拔skills

**适合场景**：

* 问题比较开放，不要求解决高复杂度和高准确率要求的问题
* 一个 Agent 就能在上下文内完成推理和执行
* 低成本验证业务价值

**不适合的场景**：

* 必须一次就完美完成的任务
* 多专业领域知识协同

![](assets/Agent%20Cookbook/file-20260316171451602.png)

### **多Agent架构**

**多 Agent 的价值在于专业分工、并行处理和复杂协作，但代价是更高的系统复杂度和 token 成本**

**适合场景**：

* 多领域复杂问题
* 需要多个方向并行探索
* **一个通用 Agent 容易被干扰、容易掉精度**
* 任务需要持续协作和综合判断



**缺点**：

* 成本会明显增加
* 调试难度更高
* **如果没有很强的可观测性，出了问题几乎很难查**

![](assets/Agent%20Cookbook/file-20260316171553763.png)

### **分层 / 监督式架构**

**Hierarchical / Supervisory**

**特点**：有一个supervisor，下面挂多specialist agent，中央控制强，职责清晰。

适合**业务流程复杂，但仍希望有中心化管控、需要审计、追踪、规则约束**的场景。

**问题**：**上下文管理复杂，中央 Agent 很容易被上下文拖垮**

![](https://rte.weiyun.baidu.com/wiki/attach/image/api/imageDownloadAddress?attachId=0b46aedd3497406d822399b2b49f2620&docGuid=ycx2-pV6pnvJpl)

### **协作式架构**

**Collaborative**

**特点**：多个 Agent 平等协作，直接通信，没有严格中心控制。

适合**研究、情报分析、探索型任务，需要多个视角交叉验证，更强调“集体智能”。**

**问题**：**通信复杂，行为容易涌现，不可预测，系统稳定性和调试难度更高**

![](assets/Agent%20Cookbook/file-20260316171719176.png)

### **顺序工作流**

**Sequential Workflow**

**特点**：按固定步骤一个接一个执行，每一步处理一个更明确的子任务

适合**可拆解为固定阶段的流程，比如草稿→审阅→发布、抽取→清洗→分析，需要稳定、可审计、可预测**


**问题**：**灵活性差，不适合需要反复回退或动态探索的任务**

![](assets/Agent%20Cookbook/file-20260316171756097.png)

### **并行工作流**

**Parallel Workflow**

**特点**：多个子任务同时跑，最后汇总结果

适合**各子问题相互独，需要不同视角同时分析，希望缩短总耗时**

**问题**：**结果冲突时需要额外聚合逻辑，不适合强依赖前后顺序的流程**

![](assets/Agent%20Cookbook/file-20260316171826560.png)
### **评估-优化循环**

**Evaluator-Optimizer**

**特点**：一个 Agent 生成结果另一个 Agent 负责评估并给反馈反复循环直到满足标准

适合**文档生成、代码生成、专业写作、复杂推理校验，“首稿不够好，但可以通过反馈逐步变好”的任务**

**问题**：**token 成本高，不适合实时性要求很高的场景**

![](assets/Agent%20Cookbook/file-20260316171914176.png)

# 架构选择

面对多种Agent架构，Anthropic给出了架构选择的一些标准，用于根据业务场景选择适合的架构。

* **可控性**：高可控场景适合使用**单 Agent / 顺序工作流**，中等可控适合使用**分层 / 监督式架构**，低可控偏探索的任务适合使用**协作式多 Agent。**
* **领域复杂度**：单领域使用单 Agent ，多领域但流程稳定使用**顺序 / 并行工作流**，多领域并且开放复杂使用**多Agent**
* **资源预算**：预算紧使用**单Agent**，上线时间紧先使用**单Agent，后续再进行演进**，长期战略建设建议从一开始就使用**模块化建设**。
* **专业能力**：单领域深专业使用**单Agent + Skill**，多专业协同使用多**Agent + Skill**。

随着Skill能力的成熟，Anthropic认为现在很多时候**不需要使用多Agent架构，而是应该先给单Agent补齐Skill能力**。Skill适合做：

* **领域知识**
* **标准化 SOP**
* **内部系统/API 接入**
* **合规规则和行业最佳实践**

# 业务思考

之前的框架选型并没有成熟的范式进行指导，导致踩了很多坑，针对这个问题，Anthropic给出了业界的标准答案，日后可以参照Anthropic提供的方式来建设Agent系统。

例如我们的一些创作场景，要求多领域的资源输入和部分可控性，就很适合使用**分层 / 监督式架构**，而一些要求高准确性、高度可控的固定流程就适合使用**顺序 / 并行工作流**来完成。

并且我们的项目一般比较紧急，可以先以单Agent进行建设，保留模块化能力，在日后慢慢演进成更通用的架构。
