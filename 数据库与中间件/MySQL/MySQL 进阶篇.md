# 存储引擎

特别注意一下，索引（Index）的概念是在引擎层去实现的，所以不同的引擎会有不同的索引机制。

![](assets/MySQL%20进阶篇/file-20260211213457369.png)
![](assets/MySQL%20进阶篇/file-20260211213512707.png)

存储引擎就是存储数据，建立索引，增删改查等技术的实现方式。**存储引擎是基于表而非数据库的**，所以又可以被叫做表类型。

```mysql
# 查询建表语句
show create table 表名;

# 查询支持的存储引擎
show engines;
```

InnoDB 作为默认引擎，支持**事务，外键和行级锁**，每张 InnoDB 引擎的表都会在磁盘中有一个对应的 .ibd 文件，存储该表的表结构，数据和索引。

MyISAM 作为早期引擎，不支持事务和外键，只支持表锁而非行级锁，访问速度较快。（目前被更流行的 NoSQL 数据库 MongoDB 取代）。

Memory 引擎，数据存储在内存中。（目前被更流行的内存数据库 Redis 取代）。

# 索引

## 索引概述

索引（Index）其实就是一种特殊的**有序的数据结构**，帮助 MySQL 高效地获取数据。在 MySQL 中，除了保存数据，数据库系统还维护着满足特定查找算法的数据结构，这种数据结构以某种方式引用数据，从而实现高效的数据查询。

**红黑树**主要用于**内存**中的查找（如编程语言的 Map/Set）。
    
**B 树 & B+ 树**主要用于**磁盘/文件系统**的查找（如数据库索引）。
    
### 红黑树

红黑树是一种**自平衡的二叉查找树（BST）**。
    
- **二叉**：每个节点最多只有两个子节点（左小右大）。
        
- **自平衡**：通过复杂的变色和旋转规则，保证树的高度不会一边倒（不像普通的二叉树可能退化成链表），从而保证查找效率是对数级别的 $O(\log N)$。
        
主要用于**内存**数据结构，例如 Java 的 `HashMap` (当链表过长时转为红黑树)，C++ STL 的 `map` 和 `set`。需要注意 JS 中的 `Map/Set` 并没有用到红黑树，而是使用的哈希表，从而确保了 $O(1)$ 的查找效率。

为什么红黑树不适合做数据库索引：
    
1. **树太“高”了**：因为它是二叉的，数据量大时，树的层级会很高。
        
2. **I/O 问题**：数据库的数据是存在磁盘上的。如果树高 20 层，查找一次可能需要读取 20 次磁盘（I/O 操作），这在物理上太慢了。我们需要一种更“矮胖”的树。
        
### B 树

为了解决红黑树“太高”的问题，B 树诞生了。它是一种**多路平衡查找树**。
    
- **多路**：一个节点可以存很多个 Key（例如 100 个），并且有多个子节点（例如 101 个），子节点数比 Key 的数量多一个。
	
- **矮胖**：因为每个节点能存很多数据，树的高度迅速降低。百万级的数据可能只需要 3 层。
	
- **数据分布**：在 B 树中，每个节点（无论是根节点、内部节点还是叶子节点）都存储真实的 Data（数据/行记录）。
        
**查找过程**：类似于翻书的目录。先在根节点找范围，然后去对应的子节点找。如果在内部节点直接找到了 Key，就直接把旁边的 Data 取出来返回。
        
### B+ 树

B+ 树是 B 树的**升级版/变体**，也是数据库最常用的结构。

**与 B 树的核心区别**：
    
1. **数据只在叶子**：非叶子节点（内部节点）**只存索引（Key）**，不存实际数据（Data）。所有的数据记录都存储在最底层的**叶子节点**上。B 树则是所有节点都可以存储数据。
	
2. **叶子相连**：所有的叶子节点通过**双向链表**连接在一起。
        
**由此带来的优势**：
    
1. **更矮**：因为内部节点不存 Data，只存 Key，所以一个磁盘页（Page）能存更多的 Key，树变得更“扁平”，磁盘 I/O 次数更少。
	
2. **查询稳定**：无论查什么数据，都必须走到最底层的叶子节点，查询时间是稳定的。

关于 Key 到底是什么东西，比如下面这条 SQL 语句：

```mysql
SELECT * FROM Users WHERE ID = 102;
```
	
这里的 `102` 就是一个 Key，用来进行比较和范围检索，因此能快速定位到数据。

### InnoDB 索引选择 B+ 树的原因

总的来说不选择 B 树的原因基本都是因为 B 树中数据散落在了整棵树中。

#### 极度适合范围查询

**场景**：`SELECT * FROM users WHERE age > 18`。
    
**B 树的做法**是，因为数据分散在各个节点，它需要通过中序遍历在树的各个层级之间**反复横跳**（回溯），极其低效。这也是 B 树最大的痛点。
    
**B+ 树的做法**是，因为所有数据都在叶子节点，且叶子节点是**有序链表**。它只需要先找到 `age=18` 这个节点，然后沿着链表**向后遍历**即可。这对于数据库的范围查找、排序、分组是非常关键的。
    
#### 扫库/扫表能力

如果你需要全表扫描，B+ 树只需要遍历底层的链表就能拿到所有数据。而 B 树需要遍历整棵复杂的树。
    
#### 磁盘预读性更好

内部节点不存数据，意味着一次 I/O 读入内存的索引页包含了更多的 Key，大大增加了命中目标 Key 的概率。
    
|**特性**|**红黑树 (Red-Black Tree)**|**B 树 (B-Tree)**|**B+ 树 (B+ Tree)**|
|---|---|---|---|
|**结构**|二叉（瘦高）|多路（矮胖）|多路（矮胖）|
|**数据存储位置**|每个节点|每个节点|**仅叶子节点**|
|**叶子节点链接**|无|无|**有 (双向链表)**|
|**查找性能**|O(log N)|O(log N)|O(log N)|
|**范围查询性能**|差 (需回溯)|差 (需回溯)|**极好 (链表遍历)**|
|**主要应用**|内存 (Map/Set)|文件系统 (部分)|**数据库索引 (MySQL)**|

### InnoDB 引擎中的具体表现

在 MySQL 的 InnoDB 引擎中，表本身就是按照 B+ 树组织的（这叫**聚簇索引**）。

1. 主键索引 (Primary Key)是这棵 B+ 树的叶子节点，存储了完整的行数据（所有列）。查询主键时，就是在这棵树上找，找到叶子就拿到了数据。
        
2. 辅助索引 (Secondary Index)：如果你给 `name` 字段建了索引，这是另一棵 B+ 树。这棵树的叶子节点，存储的不是完整数据，而是主键的值。

3. 回表 (Lookback)：当你用 `name` 查数据时，先在 `name` 树的叶子上找到对应的主键 ID，然后再拿着这个 ID 去主键索引树里找完整的行数据。

## 性能分析

可以通过如下语句查看某个数据库中各类 SQL 语句的执行频次，从而分析该数据库是以读为主还是以写为主：

```mysql
show global status like 'Com_______'; 
# 'Com_______' 代表模糊匹配的字符串
```

此外，我们还可以通过 MySQL 中的慢查询日志去分析哪些 SQL 语句的执行时间超过了我们设置的一个阈值 `long_query_time`，若超过则会记录在日志中。可以通过 MySQL 配置文件 `my.cnf` 进行设置：

```toml
slow_query_log = 1           # 开启日志
long_query_time = 1          # 超过 1 秒就记录
slow_query_log_file = /var/log/mysql/slow.log  日志存哪儿
```

# 视图 / 存储过程 / 触发器

## 视图

视图其实就是一条被保存下来的 `SELECT` 语句的结果集。它是一张**虚拟表**，本身不存数据（数据还在原来的物理表里）。

## 存储过程

允许你在数据库里写 `IF...ELSE`，写 `WHILE` 循环，声明变量，然后把一堆 SQL 语句打包成一个过程。感觉就是 SQL 中的“函数”。

```mysql
CREATE PROCEDURE UpdateUserStatus(IN userId INT)
BEGIN
    -- 这里可以写各种逻辑判断和多条 SQL
    UPDATE users SET status = 'active' WHERE id = userId;
    INSERT INTO logs (action) VALUES ('user activated');
END;
```

调用时只需要：`CALL UpdateUserStatus(1);`

使用存储过程的缺点如下：

1. **难以调试和维护：** 无法打断点，调试起来极其痛苦。
    
2. **难做版本控制：** 它是存在数据库里的，无法被 Git 管理。
    
总之知道存储过程是啥就行，现代开发中数据库只做输入数据和输出数据，数据的处理逻辑全交给业务层面去做。

## 触发器

触发器是一种特殊的存储过程。它不需要你手动 `CALL`，而是绑定在某张表上，当发生 `INSERT`、`UPDATE` 或 `DELETE` 时，自动触发执行。

典型场景：当有人向 `orders` 表插入一条数据时，自动把 `inventory` 表的库存减 1。

有啥缺点：

- 过程太过隐式，无法知道数据表的变动到底源自什么地方。
    
- 大批量导入数据时，触发器会一行一行地跟着执行，直接把数据库卡死。
