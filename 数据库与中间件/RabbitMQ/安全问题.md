RabbitMQ 只要客户端和服务端之间走的是普通 TCP，且没有启用 TLS，那理论上都可能被抓包看到认证信息和业务数据。MySQL、Redis 也有类似风险，MySQL 官方文档直接写明：未加密连接下，能访问网络的人可以观察并检查客户端和服务器之间的流量；Redis 官方文档也说明 TLS 用于保护应用和 Redis 数据库之间传输数据的隐私，并建议对敏感数据启用 TLS。来源：[MySQL 8.0 Encrypted Connections](https://dev.mysql.com/doc/refman/8.0/en/encrypted-connections.html)、[Redis Cloud database security](https://redis.io/docs/latest/operate/rc/security/database-security/)、[Redis TLS](https://redis.io/docs/latest/operate/rs/security/encryption/tls/)。

对 RabbitMQ，我建议你把规则记成一句话：

- `amqp://`：默认按明文 TCP 传输，不要用于不可信网络
- `amqps://`：AMQP over TLS，默认端口通常是 `5671`

`amqp091-go` 官方文档说明：`Dial("amqps://...")` 会启用 TLS，并使用默认的 `tls.Config`；如果你要指定私有 CA、客户端证书或更严格的校验，就用 `DialTLS`。来源：[amqp091-go 文档](https://pkg.go.dev/github.com/rabbitmq/amqp091-go)、[RabbitMQ URI spec](https://www.rabbitmq.com/docs/4.1/uri-spec)、[RabbitMQ TLS Support](https://www.rabbitmq.com/ssl.html)。  
基于这些文档可以合理推断：如果你用 `amqp://` 且认证方式是 `PlainAuth`，那应默认认为凭据和消息体都可能被抓包看到。

**RabbitMQ TLS 最小示例**

如果服务端已经启用了 TLS，且证书由系统信任的 CA 签发，最简单是直接用 `amqps://`：

```go
package main

import (
	"log"

	amqp "github.com/rabbitmq/amqp091-go"
)

func main() {
	conn, err := amqp.Dial("amqps://user:password@rabbitmq.example.com:5671/")
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatal(err)
	}
	defer ch.Close()

	log.Println("connected to RabbitMQ over TLS")
}
```

如果你用的是自签 CA，或者想显式配置 TLS，建议用 `DialTLS`：

```go
package main

import (
	"crypto/tls"
	"crypto/x509"
	"log"
	"os"

	amqp "github.com/rabbitmq/amqp091-go"
)

func main() {
	caPEM, err := os.ReadFile("certs/ca.pem")
	if err != nil {
		log.Fatal(err)
	}

	roots := x509.NewCertPool()
	if !roots.AppendCertsFromPEM(caPEM) {
		log.Fatal("failed to parse CA cert")
	}

	tlsConfig := &tls.Config{
		MinVersion: tls.VersionTLS12,
		RootCAs:    roots,
		ServerName: "rabbitmq.example.com",
	}

	conn, err := amqp.DialTLS(
		"amqps://user:password@rabbitmq.example.com:5671/",
		tlsConfig,
	)
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	ch, err := conn.Channel()
	if err != nil {
		log.Fatal(err)
	}
	defer ch.Close()

	log.Println("connected to RabbitMQ over TLS with custom CA")
}
```

如果服务端要求双向 TLS，再加客户端证书：

```go
package main

import (
	"crypto/tls"
	"crypto/x509"
	"log"
	"os"

	amqp "github.com/rabbitmq/amqp091-go"
)

func main() {
	caPEM, err := os.ReadFile("certs/ca.pem")
	if err != nil {
		log.Fatal(err)
	}

	clientCert, err := tls.LoadX509KeyPair("certs/client.crt", "certs/client.key")
	if err != nil {
		log.Fatal(err)
	}

	roots := x509.NewCertPool()
	if !roots.AppendCertsFromPEM(caPEM) {
		log.Fatal("failed to parse CA cert")
	}

	tlsConfig := &tls.Config{
		MinVersion:   tls.VersionTLS12,
		RootCAs:      roots,
		Certificates: []tls.Certificate{clientCert},
		ServerName:   "rabbitmq.example.com",
	}

	conn, err := amqp.DialTLS(
		"amqps://user:password@rabbitmq.example.com:5671/",
		tlsConfig,
	)
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	log.Println("mTLS connection established")
}
```

这几个点不要省：

- `ServerName` 要和服务端证书里的域名一致
- 不要随手设置 `InsecureSkipVerify: true`
- 生产环境至少要求 `TLS 1.2+`
- 账号密码不要硬编码，放环境变量或 secret manager

**MySQL、Redis 会不会一样**

会，原则完全一样：

- RabbitMQ 未加密：可能泄露用户名、密码、消息体
- MySQL 未加密：可能泄露用户名、密码、SQL、查询结果
- Redis 未加密：可能泄露密码、命令、缓存内容

只是表现形式不同，本质都是“应用层协议跑在未加密 TCP 上”。

**Go 里常见安全连接写法**

RabbitMQ：`amqps://...` 或 `amqp.DialTLS(...)`

MySQL：驱动层注册 TLS 配置后再用 DSN 里的 `tls=` 参数  
常见是 `github.com/go-sql-driver/mysql`

```go
mysql.RegisterTLSConfig("custom", &tls.Config{...})
db, err := sql.Open("mysql", "user:pass@tcp(db.example.com:3306)/app?tls=custom")
```

Redis：客户端配置 `TLSConfig`  
常见是 `github.com/redis/go-redis/v9`

```go
rdb := redis.NewClient(&redis.Options{
    Addr: "redis.example.com:6380",
    TLSConfig: &tls.Config{
        MinVersion: tls.VersionTLS12,
        ServerName: "redis.example.com",
    },
})
```

**实战判断标准**

- `localhost` 单机开发：风险相对低，但不是零
- 同一台宿主机不同容器：仍建议考虑 TLS 或至少网络隔离
- 办公网、云内网、跨可用区、跨机房、公网：默认必须上 TLS
- “内网所以安全” 这个假设通常不可靠
