# gRPC

gRPC 默认使用 Protobuf 作为其接口定义语言和底层数据序列化格式。开发者首先需要编写 `.proto` 文件，在这个文件中定义服务（Service）的方法以及请求/响应的消息结构。与 JSON 或 XML 这类基于文本的格式不同，Protobuf 会将数据序列化为紧凑的二进制格式。它通过给每个字段分配一个唯一的 Tag 编号，并在传输时省去字段名，从而极大压缩了数据体积，解析速度也远超 JSON。通过 `protoc` 编译器，一套 `.proto` 文件可以自动生成各种主流语言的数据结构和路由代码。

gRPC 的高性能在很大程度上归功于其底层绑定的 HTTP/2 协议，而不是传统的 HTTP/1.1。

- **多路复用：** 在 HTTP/1.1 中，并发请求通常需要建立多个 TCP 连接（存在队头阻塞问题）。HTTP/2 引入了“流（Stream）”的概念，允许在单一的 TCP 连接上并发交错发送多条请求和响应消息，极大地降低了连接建立的开销。
- **流式通信：** 得益于 HTTP/2，gRPC 不仅支持传统的“一问一答”，还原生支持三种流式通信：
    - **服务端流：** 客户端发送一个请求，服务端返回一个数据流（适用于持续推送数据）。  
    - **客户端流：** 客户端发送一个数据流，服务端返回一个响应。
    - **双向流：** 双方可以同时向对方发送数据流（例如实时的状态同步）。       
- **头部压缩（HPACK）：** HTTP/2 对 Header 进行了压缩，并且通信双方会维护一个 Header 字典，避免了每次请求都发送重复的元数据。
    

gRPC 隐藏了网络通信的复杂细节，这是通过“存根（Stub）”机制实现的。

- **客户端 Stub：** 当客户端调用一个方法时，实际上是调用了本地生成的 Stub 对象。Stub 负责将调用的参数使用 Protobuf 序列化，将其封装为 HTTP/2 帧，并发送给目标服务器。    
- **服务端 Stub：** 服务端收到网络字节流后，底层的 gRPC 框架会将其解包、反序列化，然后路由到开发者实际编写的代码逻辑中。执行完毕后，再按原路打包返回给客户端。
    

一次完整的调用流程：
1. **客户端发起调用：** 像调用本地方法一样调用 Client Stub。
2. **编码与打包：** Stub 将请求参数序列化为 Protobuf 二进制格式。
3. **网络传输：** gRPC 核心库将二进制数据映射为 HTTP/2 数据帧，通过 TCP 连接发送。
4. **服务端接收：** 服务端的 gRPC Server 接收帧，组装数据并进行反序列化。
5. **业务处理：** 将反序列化后的参数传递给具体的 Service 实现类进行处理。
6. **返回结果：** 处理结果再次经过序列化，通过 HTTP/2 返回给客户端。

# ES

ES 是一个基于 Apache Lucene 构建的、分布式的、RESTful 风格的实时搜索与数据分析引擎。其核心设计目标是为了解决海量全文本数据的低延迟检索与高并发分析难题。


倒排索引是 Elasticsearch 实现高性能全文检索的第一性原理。传统关系型数据库通常采用正向索引（如 B+ 树），通过主键或行 ID 查找具体字段内容；而倒排索引则完全反转了这一过程。

* **分词处理：** 当一条文档写入 ES 时，系统会首先通过分词器将其长文本切分为一系列独立的、具备语义的词元（Term）。
* **构建索引表：** ES 会维护一个核心字典结构。字典的键（Key）为全局唯一的词元（Term），值（Value）则是包含该词元的文档 ID 列表，同时还会记录词频（TF）和位置信息（Position）。
* **检索效率：** 在查询时，系统无需进行全表扫描，而是通过对查询词分词后，直接在倒排字典中进行 $O(1)$ 或 $O(\log n)$ 复杂度的查找，瞬间获取命中该词元的文档集合，从而支撑起海量文本下的即时响应。

为了克服单机 Lucene 实例在存储容量与计算性能上的物理瓶颈，ES 引入了分布式外壳，其组件设计与主流分布式流处理系统（如 Kafka）存在高度的拓扑相似性：

* **索引 (Index)：** 逻辑上的数据分类实体，类似于关系型数据库中的表（Table）或 Kafka 中的主题（Topic）。
* **分片 (Shard)：** 物理上的基本存储与计算单元。一个索引内部的数据会被水平切分为多个分片。**每个分片在底层本质上都是一个完整且独立的 Apache Lucene 引擎实例**，拥有自己独立的倒排索引文件。
* **主分片与副本分片 (Primary / Replica Shard)：** 
	* Primary Shard（主分片）： 负责承载初始的文档写入请求，并将数据同步复制给其对应的副本分片。
	* Replica Shard（副本分片）： 作为主分片的冗余备份，不直接主导写入，但可以分担集群的读取请求，并在主分片所在节点宕机时参与选主升级。
* **主节点 (Master Node)：** 负责管理集群级别的元数据状态（如索引的创建与删除、分片在物理节点间的动态分配与迁移）。现代 ES 采用内建的分布式共识协议进行 Master 选举，不再依赖外部协调组件。

假设现有 3 台物理服务器构成一个 ES 集群（Node 1, Node 2, Node 3）。现创建一个名为 `user_logs` 的索引，配置参数为：**3 个主分片（P0, P1, P2），副本因子为 1（即每个主分片各有一个物理副本 R0, R1, R2）**。

集群主节点会通过内置调度算法，将这 6 个分片均匀且交错地散布在 3 台机器上：
* **Node 1 进程：** 托管 P0（主分片）与 R2（副本分片）
* **Node 2 进程：** 托管 P1（主分片）与 R0（副本分片）
* **Node 3 进程：** 托管 P2（主分片）与 R1（副本分片）

当客户端发起一条写入文档请求（例如插入一条用户日志）时，ES 会根据文档的唯一标识 `_id` 实施动态路由，公式为：
$$\text{Shard} = \text{hash}(\text{\_id}) \pmod{\text{NumberOfPrimaryShards}}$$
由于主分片数（这里为 3）在创建索引时已确定，该公式能够确保数据绝对均匀地散落在 P0、P1 或 P2 中。请求被路由到对应主分片所在的 Node 执行写入，成功后再由该 Node 将数据并发推送到对应副本分片所在的 Node，完成物理落盘。

当客户端发起全文检索时，查询请求会并发分发到 P0、P1、P2（或其对应的 Replica）上并行计算，各分片独立检索自己的 Lucene 倒排索引，最后由协调节点合并结果返回，实现了算力的横向叠加。

若 Node 1 发生物理断电，集群会立即感知。由于 P0 丢失，Master 节点会迅速将位于 Node 2 上的 R0 升级为新的主分片。在选举与恢复期间，Node 2 和 Node 3 上的 P1、P2 依然保持完全可用，单点故障的爆炸半径被严格限制在受损分片范围内。

区别于需要专用 TCP 客户端的中间件，ES 暴露了纯粹的 JSON over HTTP 接口。无论是结构映射（Mapping）定义、基于 DSL（Domain Specific Language）的复杂多维聚合查询，还是集群状态监控，均通过标准 HTTP 方法（GET/POST/PUT/DELETE）配合 JSON 载荷完成，具备极高的通用性与低接入成本。

随着人工智能及大语言模型（LLM）生态的发展，现代 ES 已演进为高性能的向量数据库。它不仅支持传统的基于词频（如 BM25 算法）的文本匹配，还原生支持存储多维浮点数稠密向量，并内置了高效的 kNN（K-最近邻）搜索算法。这使得 ES 能够越过字面限制，进行深度的语义模糊检索，成为当前构建 RAG（检索增强生成）系统与 Agent 记忆模块的核心基础设施。

# Kafka


Kafka 是一个分布式的、持久化的、追加写入的提交日志系统。现代 Kafka 采用 KRaft 架构替代了外部依赖（ZooKeeper）。KRaft 基于 Raft 共识算法，在 Controller 节点之间维护高可用的元数据状态机，通过日志复制和 Leader 选举独立完成集群共识。

**Kafka 主要的使用场景一个就是高并发场景下缓解数据库压力，削峰填谷；另一个就是用作日志链路缓冲日志流，让ES可以慢慢消费。**

一些概念：

* Topic：纯逻辑概念，代表一类消息的集合，无对应的单一物理文件。
* Partition：物理存储与扩展的核心单元。一个 Topic 被划分为多个 Partition，每个 Partition 在物理磁盘上对应独立的日志目录（包含 `.log` 和 `.index` 等文件）。Kafka 仅保证单一 Partition 内部的严格顺序性。
* Offset：Partition 内消息的单调递增序号。由消费者端（或 Consumer Group）主动维护和提交，使得 Broker 端维持无状态的高效读写。
* Consumer Group：通过组内消费者实例对 Partition 的独占消费机制，实现了发布-订阅模型与点对点队列模型的统一。

以 3 个 Broker 节点、1 个 Topic（设为 3 个 Partition: P0, P1, P2）、ReplicationFactor 为 2 的集群为例：

* **Broker 进程**：每个节点上独立运行的 Kafka Java 进程。多个 Broker 通过共享集群配置与网络通信构成完整集群。
* **副本分布**：为保证容错，同一 Partition 的不同副本严格分布在不同的 Broker 上。Leader 唯一负责处理该 Partition 所有客户端读写请求的节点。Follower 不直接对接客户端，通过异步拉取（Fetch）操作从 Leader 同步增量日志，维持数据一致性。
* **状态示例**：
    * Broker 0: 承载 P0 (Leader), P2 (Follower)
    * Broker 1: 承载 P1 (Leader), P0 (Follower)
    * Broker 2: 承载 P2 (Leader), P1 (Follower)

将单一主题的请求分散到多个独立的 Partition 复制组中，具备以下工程优势：

* **极致的负载均衡**：通过将不同 Partition 的 Leader 节点均匀离散至集群的各个 Broker 上，避免了单一节点的网络 I/O 与磁盘读写成为系统全局瓶颈。
* **控制故障爆炸半径**：当单一物理节点（如 Broker 0）发生宕机时，仅影响以该节点为 Leader 的特定 Partition（如 P0）。其余 Leader 位于健康节点的 Partition 读写不受影响。集群控制器会迅速在存活的 Follower 中触发 Leader Election 恢复服务，保障系统的高可用性。

一些简单的代码例子：

producer：
```go
package main

import (
	"context"
	"fmt"
	"log"
	"time"

	"github.com/segmentio/kafka-go"
)

func main() {
	// 定义要写入的 Topic 和 Kafka 集群地址
	topic := "UserActivity"
	brokers := []string{"localhost:9092"} // 实际生产环境中通常会填入多个 Broker 地址

	// 初始化一个 kafka.Writer
	// Writer 内部维护了连接池，并自动处理 Partition 路由和失败重试
	w := &kafka.Writer{
		Addr:                   kafka.TCP(brokers...),
		Topic:                  topic,
		Balancer:               &kafka.Hash{}, // 路由策略：根据 Key 的 Hash 值决定写入哪个 Partition
		RequiredAcks:           kafka.RequireAll, // 对应 acks=all，要求所有 ISR 副本确认，保证最高的数据可靠性
		AllowAutoTopicCreation: true,          // 如果 Topic 不存在则自动创建（生产环境通常关闭）
	}

	defer func() {
		if err := w.Close(); err != nil {
			log.Fatal("failed to close writer:", err)
		}
	}()

	// 模拟连续发送 5 条消息
	for i := 0; i < 5; i++ {
		msgKey := fmt.Sprintf("user-%d", i)
		msgValue := fmt.Sprintf("click-event-timestamp-%d", time.Now().UnixNano())

		err := w.WriteMessages(context.Background(),
			kafka.Message{
				Key:   []byte(msgKey),   // Key 决定了消息落入哪个具体的 Partition
				Value: []byte(msgValue), // 实际的业务数据载荷
			},
		)

		if err != nil {
			log.Printf("写入消息失败: %v\n", err)
		} else {
			fmt.Printf("成功写入消息 | Key: %s | Value: %s\n", msgKey, msgValue)
		}
		time.Sleep(time.Second)
	}
}
```

consumer：
```go
package main

import (
	"context"
	"fmt"
	"log"

	"github.com/segmentio/kafka-go"
)

func main() {
	topic := "UserActivity"
	brokers := []string{"localhost:9092"}
	groupID := "analytics-service-group" // 声明当前消费者属于哪个组

	// 初始化一个 kafka.Reader
	r := kafka.NewReader(kafka.ReaderConfig{
		Brokers:  brokers,
		GroupID:  groupID,
		Topic:    topic,
		MinBytes: 10e3, // 10KB：每次向 Broker 拉取的最小数据量，达不到则等待，优化网络开销
		MaxBytes: 10e6, // 10MB：每次拉取的最大数据量
	})

	defer func() {
		if err := r.Close(); err != nil {
			log.Fatal("failed to close reader:", err)
		}
	}()

	fmt.Printf("消费者已启动，加入组 [%s]，正在监听 Topic [%s]...\n", groupID, topic)

	// 开启一个无限循环，持续拉取消息
	for {
		// ReadMessage 会阻塞，直到有新消息到达
		// 它在内部自动处理了 Offset 的提交 (Auto Commit)
		m, err := r.ReadMessage(context.Background())
		if err != nil {
			log.Printf("读取消息出错: %v\n", err)
			break
		}

		// 打印收到的消息元数据和内容
		fmt.Printf("收到消息 | Partition: %d | Offset: %d | Key: %s | Value: %s\n",
			m.Partition, // 这条消息物理上存在哪个分区
			m.Offset,    // 在该分区日志文件中的绝对单调递增序号
			string(m.Key),
			string(m.Value),
		)
	}
}
```

- **路由与负载均衡 (`Balancer: &kafka.Hash{}`)：** 在 Producer 中设置了 Hash 路由。这意味着具有相同 Key（如同一个 `user_id`）的消息，其 Hash 值计算结果相同，会永远被路由到同一个 Partition 中，这保证了**局部有序性**。
    
- **安全性保障 (`RequiredAcks: kafka.RequireAll`)：** 这行代码直接呼应了底层架构中的 ISR（In-Sync Replicas）机制。它告诉 Broker 的 Leader：“收到这条消息后不要立刻返回成功，必须等后台的 Follower 们都把这条日志同步到它们各自的 `.log` 文件后，再告诉客户端写入成功。”
    
- **游标控制 (`ReadMessage` 中的 Offset)：** 当 Consumer 读取消息时，它并不修改服务端的 `.log` 文件（这是追加写入的只读模型）。Consumer 只是拿到了当前消息的 `Offset`，并在后台将其提交到 Kafka 内部的特殊 Topic（`__consumer_offsets`）中，以此来记录当前消费组在这个 Partition 的“阅读进度”。

此外 Kafka 还做了如下优化：

* **顺序磁盘 I/O**：采用追加模式，规避磁盘随机寻道时间，最大化底层存储设备的写入吞吐率。
* **Page Cache**：将内存管理职责下移至操作系统内核（页缓存），减少 JVM 堆内存压力与垃圾回收（GC）引起的系统停顿。
* **零拷贝 (Zero-Copy)**：在消费者拉取数据时，调用操作系统的 `sendfile` 指令，使数据流直接从 Page Cache 经过网络接口卡（NIC）传输，省去了内核态到用户态的数据拷贝开销。

# Asynq

Redis 是一个基于内存的数据结构存储系统，通常被用作数据库、缓存和消息代理，提供了 String、List、Set、Hash、Sorted Set (Zset) 等数据结构。比如，我们可以用 List 的 `LPUSH` 和 `BRPOP` 可以实现简单的阻塞队列；Sorted Set 可以通过分数（Score）来对任务进行排序，非常适合实现延时任务。如果直接用 Redis 的 List 来做任务队列，需要自己处理非常多复杂的边缘情况：任务失败了怎么办？如何实现重试逻辑？怎么做延时任务？进程崩溃了正在执行的任务会不会丢失？

**Asynq** 就是为了解决这些工程痛点而诞生的。它是一个用 Go 语言编写的、依赖 Redis 作为后端的分布式任务队列库。

**Asynq 提供了：**
1. **可靠性保证：** Asynq 保证任务至少被成功处理一次。如果一个 Worker 处理任务时崩溃，任务会被重新放回队列。
2. **任务调度：** 可以轻松实现“延迟 10 分钟执行”或“每天凌晨 2 点执行”的定时任务。
3. **重试机制与退避策略：** 任务失败后，Asynq 会自动重试，并且重试间隔会逐渐拉长，避免雪崩效应。
4. **死信队列：** 彻底失败（达到最大重试次数）的任务会被转移到特定队列，等待人工介入排查，而不是无限循环阻塞系统。
5. **任务唯一性与聚合：** 可以防止相同的任务被重复入队，或者将短时间内产生的多个同类任务合并为一个处理。

Asynq 的巧妙之处在于它深度利用了 Redis 的多种数据结构来管理任务的生命周期（状态机）：

- **Pending（待执行）：** 存放在 Redis 的 **List** 中。Worker 通过阻塞命令（如 `BRPOP`）监听这个 List，一旦有任务就立刻拉取，保证极低的延迟。
- **Scheduled / Retry（计划执行 / 等待重试）：** 存放在 Redis 的 **Sorted Set (Zset)** 中。时间戳作为 Score。一个后台协程会不断轮询，把到了时间的任务从 Zset 迁移到 Pending List 中。
- **Active（正在执行）：** 当 Worker 拿走任务后，任务会被移入另一个 List 或 Set，以标记正在执行。如果 Worker 超时未报告心跳，Asynq 会认为 Worker 挂了，从而将任务重新捞回队列。
- **任务元数据：** 任务的具体 payload（例如一段 JSON 数据）、重试次数、错误信息等，通常使用 Redis 的 **Hash** 结构存储。

一些使用 Asynq 的场景：

1. **耗时的 I/O 或计算操作：** 比如生成复杂的空间数据分析报告、处理大文件上传。前端发起请求，后端只需将任务推给 Asynq 然后立即返回 "202 Accepted"，前端通过长连接（如 SSE 或 WebSocket）监听状态，保持界面的响应性。
2. **可靠的第三方 API 调用：** 发送邮件、短信，或者调用外部支付网关。利用 Asynq 的重试机制应对网络抖动和第三方服务的不稳定。

和 Kafka 等消息队列的区别对比：
```text
Kafka:
  关注事件流(比如一个“下单”下单发生了)
  多个 consumer group 可以各自消费同一批消息
  适合“发生了什么”
  通过 offset 记录消费进度

Asynq:
  关注任务执行(比如设置15分钟超时，那么支付创建后就要投递一个15分钟后发生的取消订单的异步任务)
  一个任务通常只需要一个 worker 执行
  适合“去做什么”
  通过任务状态管理 pending / scheduled / active / retry / archived
```

以下是一个简单的例子：

tasks.go
```go
package tasks

import (
	"encoding/json"
	"github.com/hibiken/asynq"
)

// 1. 定义任务的唯一类型标识
const (
	TypeEmailDelivery = "email:deliver"
)

// 2. 定义任务需要的数据 (Payload)
type EmailDeliveryPayload struct {
	UserID     int    `json:"user_id"`
	TemplateID string `json:"template_id"`
}

// 3. 封装一个快捷创建任务的函数
func NewEmailDeliveryTask(userID int, templateID string) (*asynq.Task, error) {
	payload, err := json.Marshal(EmailDeliveryPayload{
		UserID:     userID,
		TemplateID: templateID,
	})
	if err != nil {
		return nil, err
	}
	// asynq.NewTask 接收类型和序列化后的 Payload
	return asynq.NewTask(TypeEmailDelivery, payload), nil
}
```

```go
package main

import (
	"log"
	"time"
	"your_project/tasks" // 替换为你的实际包路径

	"github.com/hibiken/asynq"
)

func main() {

	client := asynq.NewClient(asynq.RedisClientOpt{Addr: "127.0.0.1:6379"})
	defer client.Close()

	// 1. 创建一个任务实例
	task, err := tasks.NewEmailDeliveryTask(42, "welcome_email_v1")
	if err != nil {
		log.Fatalf("无法创建任务: %v", err)
	}

	// 2. 将任务推入队列 (Enqueue)
	// 在这里可以配置 Asynq 的高级特性：
	// - asynq.MaxRetry(5): 最多重试 5 次
	// - asynq.ProcessIn(10 * time.Minute): 延迟 10 分钟后执行
	info, err := client.Enqueue(
		task,
		asynq.MaxRetry(5),
		asynq.Timeout(2*time.Minute), // 限制 worker 处理这个任务的最大时间
	)
	
	if err != nil {
		log.Fatalf("任务入队失败: %v", err)
	}
	
	log.Printf("成功将任务放入队列: ID=%s, 队列=%s", info.ID, info.Queue)
}
```

```go
	// 1. 配置并启动 Asynq Server (Worker)
	srv := asynq.NewServer(

	log.Println("Worker 已启动，正在等待任务...")

	}
```
