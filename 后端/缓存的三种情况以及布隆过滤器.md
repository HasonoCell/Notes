### 一、 缓存三大经典问题

#### 1. 缓存穿透

* **概念：** 客户端请求的数据，在 Redis 中**不存在**，在数据库中也**绝对不存在**。
* **现象：** 因为缓存没命中，请求会打到数据库；数据库查不到，自然也不会写入缓存。结果就是，每一次这样的请求都会无情地穿透 Redis，直接暴击数据库。如果黑客伪造大量不存在的 ID（比如 `id = -1`）发起疯狂请求，数据库会瞬间宕机。
* **解决方案：**
1. **缓存空对象（Null Object）：** 哪怕数据库返回空，也把这个空结果（比如 `{}` 或 `null`）存入 Redis，设置一个较短的过期时间（TTL）。下次同样的非法请求来，Redis 直接返回空。
2. **布隆过滤器（Bloom Filter）：** （下文详解）在请求到达 Redis 之前，先过一道布隆过滤器。过滤器会告诉你这个 ID “绝不可能存在”，直接拒绝请求。

#### 2. 缓存击穿 

* **概念：** 缓存中**有**数据，数据库中也**有**数据，但这通常是一个**极度热点（Hotspot）的单点 Key**（比如微博热搜第一名、秒杀商品）。在这个 Key 的过期时间（TTL）刚到、失效的那一瞬间，成千上万的并发请求同时涌来。
* **现象：** 第一个请求发现缓存没数据，去查数据库。但是在它把数据重新写回 Redis 之前，剩下的 9999 个请求也发现缓存没数据，也跟着去查数据库。巨大的流量像一根针一样，击穿了这一个点的缓存，压垮数据库。
* **解决方案：**
1. **互斥锁（Mutex Lock / 分布式锁）：** 发现缓存未命中时，不立刻查 DB，而是先尝试获取锁。拿到锁的那个线程去查 DB 并重建缓存，其他拿不到锁的线程原地等待（Sleep），醒来后直接读刚重建好的缓存。
2. **逻辑过期：** 缓存本身不设置物理 TTL（永不过期），但在 Value 中存一个“逻辑过期时间戳”。发现逻辑过期后，返回旧数据，同时开启一个异步后台协程去更新缓存。

#### 3. 缓存雪崩

* **概念：** 缓存中**有**大量数据。但是在某一时刻，**大批量的 Key 同时过期**，或者 **Redis 节点直接宕机**了。
* **现象：** 击穿是“单点爆发”，雪崩是“全面崩溃”。大量请求同时找不到缓存，全部涌向数据库，引发系统级雪崩。
* **解决方案：**
1. **TTL 随机化：** 坚决避免大量 Key 使用相同的过期时间。在基础过期时间上，加上一个随机的抖动值（Jitter），比如 `TTL = 10分钟 + Random(1~3分钟)`。
2. **Redis 高可用架构：** 部署哨兵模式（Sentinel）或集群模式（Cluster），防止单节点宕机。
3. **限流降级：** 实在扛不住了，在网关层直接丢弃部分请求，或返回默认的兜底页面，保护核心数据库。

### 二、 布隆过滤器 (Bloom Filter)

为了解决“缓存穿透”问题，我们需要一种能**极速判断一个元素存不存在，且占用内存极小**的数据结构。这就是由 Burton Howard Bloom 在 1970 年发明的布隆过滤器。

#### 核心原理

布隆过滤器本质上是一个非常长的**二进制位数组**，外加 $k$ 个独立的**哈希函数**。

1. **初始化：** 一个包含 $m$ 位的数组，所有位初始都是 `0`。
2. **添加元素：** 当加入一个字符串（比如 "apple"）时，用 $k$ 个哈希函数对它计算，得到 $k$ 个位置索引，将数组中这 $k$ 个位置的位设置为 `1`。
3. **查询元素：** 当检查一个字符串是否存在时，同样用这 $k$ 个哈希函数算一遍。
* 如果算出来的 $k$ 个位置中，有任何一个位是 `0`，那么这个字符串绝对不存在。
* 如果算出来的 $k$ 个位置全都是 `1`，那么这个字符串可能存在（因为这些 `1` 可能是其他字符串添加时顺手改的，这叫“哈希冲突”或“假阳性 / False Positive”）。

总之就是布隆过滤器说你不在，你就肯定不在；说你在，你只是有可能在。

### 三、 Go 语言手写布隆过滤器

在实际生产中，后端通常会直接使用 Redis 的 `RedisBloom` 模块。但为了深入理解它的底层，我们可以用 Go 语言自己实现一个。

在 Go 中，我们没有原生的 `bit` 类型，最小的类型是 `byte` (8位)。为了更高效，我们通常用 `uint64` (64位) 的切片（Slice）来拼接出一个超长的位图。

下面这段代码包含了详尽的位运算注释。

```go
package main

import (
	"fmt"
	"hash/fnv"
)

// BloomFilter 布隆过滤器结构体
type BloomFilter struct {
	// bitSet 是底层的位数组。
	// 为什么用 []uint64？因为一个 uint64 包含 64 个 bit。
	// 如果我们需要 6400 个 bit，只需要一个长度为 100 的 []uint64 切片即可。
	bitSet []uint64
	// size 表示布隆过滤器的总 bit 位数 (即公式中的 m)
	size uint
	// hashCount 表示哈希函数的个数 (即公式中的 k)
	hashCount uint
}

// NewBloomFilter 初始化布隆过滤器
func NewBloomFilter(size uint, hashCount uint) *BloomFilter {
	// 计算需要多少个 uint64。向上取整。
	// 例如：size=100，100/64 = 1，余数不为0，所以需要 2 个 uint64。
	sliceLen := size / 64
	if size%64 != 0 {
		sliceLen++
	}

	return &BloomFilter{
		bitSet:    make([]uint64, sliceLen),
		size:      size,
		hashCount: hashCount,
	}
}

// getHashIndices 获取一个字符串的 k 个哈希索引位
func (bf *BloomFilter) getHashIndices(data string) []uint {
	indices := make([]uint, bf.hashCount)
	
	// 这里为了演示简便，我们使用 Go 标准库的 FNV 哈希算法。
	// 在生产环境中，通常使用 MurmurHash3，并只算两次哈希，通过 h1 + i*h2 来模拟 k 次哈希以提升性能。
	for i := uint(0); i < bf.hashCount; i++ {
		h := fnv.New64a()
		// 每次哈希附加不同的盐值 (i) 来模拟不同的哈希函数
		h.Write([]byte(fmt.Sprintf("%d-%s", i, data)))
		// 取模，保证索引落在 0 到 size-1 的范围内
		indices[i] = uint(h.Sum64()) % bf.size
	}
	return indices
}

// Add 添加元素到布隆过滤器
func (bf *BloomFilter) Add(data string) {
	indices := bf.getHashIndices(data)
	for _, index := range indices {
		// 核心位运算：定位具体是哪个 bit
		// 1. index / 64 找到它属于 slice 中的哪一个 uint64 块
		sliceIndex := index / 64
		// 2. index % 64 找到它在这个 uint64 块内部的偏移量 (0-63)
		bitOffset := index % 64

		// 3. 将 1 左移 bitOffset 位，然后用按位或 (|) 操作，将对应位置为 1
		// 举例：如果 bitOffset 是 2，1 << 2 变成二进制 00000100
		// 任何数与它按位或，都会把这一位强制变成 1，其他位不变。
		bf.bitSet[sliceIndex] |= (1 << bitOffset)
	}
}

// Contains 检查元素是否（可能）存在
func (bf *BloomFilter) Contains(data string) bool {
	indices := bf.getHashIndices(data)
	for _, index := range indices {
		sliceIndex := index / 64
		bitOffset := index % 64

		// 核心位运算：检查某个 bit 是否为 1
		// 使用按位与 (&) 操作
		// 如果对应位是 1，(某数 & 00000100) 的结果就不会是 0。
		// 如果对应位是 0，(某数 & 00000100) 的结果必然是 0。
		if (bf.bitSet[sliceIndex] & (1 << bitOffset)) == 0 {
			// 只要发现有一个位是 0，绝对不存在，立刻返回 false
			return false
		}
	}
	// 所有位都是 1，可能存在
	return true
}

func main() {
	// 初始化一个容量为 1000 个 bit，使用 3 个哈希函数的布隆过滤器
	bf := NewBloomFilter(1000, 3)

	// 模拟写入存在数据库中的有效 ID
	validKeys := []string{"user:101", "user:102", "user:888"}
	for _, key := range validKeys {
		bf.Add(key)
		fmt.Printf("已添加有效数据: %s\n", key)
	}

	fmt.Println("--------------------")

	// 模拟查询
	testKeys := []string{"user:101", "user:999"}

	for _, key := range testKeys {
		if bf.Contains(key) {
			// 过滤器说存在，放行请求，去查 Redis/DB
			fmt.Printf("请求 %s: 布隆过滤器说【可能存在】，放行请求去查数据库。\n", key)
		} else {
			// 过滤器说不存在，直接在网关层拦截，保护后端！
			fmt.Printf("请求 %s: 布隆过滤器说【绝对不存在】，直接丢弃请求！(成功阻挡缓存穿透)\n", key)
		}
	}
}

```
