认证 (AuthN)： 证明“你是谁”。
授权 (AuthZ)： 决定“你能做什么”。

在 HTTP 状态码中，它们也有明确的对应关系：

* **401 Unauthorized:** 认证失败。
* **403 Forbidden:** 授权失败。

## 一、 认证 (Authentication)

认证的核心目标是验证用户的身份凭证（Credentials）。常见的凭证包括密码、手机验证码、生物识别（指纹/FaceID）或物理密钥。

### 常见的认证实现方式：

1. **Session / Cookie (有状态认证)**
* **原理：** 用户登录成功后，服务器在内存或 Redis 中生成一个 Session，并给前端返回一个 Session ID（通常种在 Cookie 里）。之后的每次请求，前端都会携带这个 Cookie，服务器去比对寻找对应的 Session。
* **优点：** 服务器对登录状态有绝对的控制权，想踢谁下线直接删掉后端的 Session 即可。
* **缺点：** 它是“有状态”的。在分布式系统中，如果用户第一次请求打到了服务器 A，第二次打到了服务器 B，就会出现找不到 Session 的问题（除非用 Redis 做集中式 Session 存储）。


2. **JWT (JSON Web Token) (无状态认证)**
* **原理：** 登录成功后，服务器用自己的私钥将用户信息（如 UserID）签名，生成一个长长的字符串（Token）发给前端。前端把 Token 存在 LocalStorage 中，后续请求放在 HTTP Header 的 `Authorization: Bearer <token>` 里。服务器收到请求后，用公钥/密钥校验签名是否被篡改，校验通过即可信任该身份。
* **优点：** 极其适合微服务和分布式架构。服务器不需要存储状态，只要 CPU 算力进行签名校验即可。
* **缺点：** Token 签发后在过期前很难被主动撤销（除非引入黑名单机制，但这又破坏了无状态的初衷）。


3. **OAuth 2.0 / OIDC (第三方统一认证)**
* **原理：** 你不想自己管密码，于是让用户点击“使用 Google/GitHub/微信 登录”。你的系统信任这些大厂的认证结果。
* **场景：** 适用于需要快速接入第三方生态，或者企业内部建设统一的 SSO（单点登录）网关。

## 二、 授权 (Authorization)

当系统通过 AuthN 明确了“你是 User 123”之后，接下来的工作交由 AuthZ 接管。授权的核心是评估该用户是否有权限对特定的**资源**执行特定的**操作**。

### 常见的授权模型与实现方式：

1. **ACL (Access Control List - 访问控制列表)**
* **原理：** 最原始的方式。直接在资源上挂一个名单。比如“文件 A 的允许访问名单：张三、李四”。
* **缺点：** 用户量一大，管理起来就是灾难。


2. **RBAC (Role-Based Access Control - 基于角色的访问控制)**
* **原理：** 引入“角色”作为解耦层。给用户分配角色（如 `Admin`, `Editor`, `Viewer`），再把权限分配给角色。
* **实现：** 在 Go 或 Java 等后端的工程实践中，通常会引入像 **Casbin** 这样的规则引擎。开发者只需要编写好包含 `g(用户, 角色)` 和 `p(角色, 资源, 动作)` 的配置文件，引擎会自动完成匹配运算。这也是目前绝大多数中小型系统的标配。


3. **ABAC (Attribute-Based Access Control - 基于属性的访问控制)**
* **原理：** 极其精细但复杂的模型。它不看角色，而是看属性的布尔逻辑。比如判断条件为：“当 `用户的部门 == 财务部` 且 `当前时间在 9:00-18:00 之间` 且 `请求的账单金额 < 10000` 时，允许访问”。
* **场景：** 适用于金融、医疗或政务等对权限粒度要求极其苛刻的复杂业务系统。

## 三、 它们在实际系统中的协同工作流

在一个标准的现代 Web 架构中，认证和授权通常是流水线上的两道关卡（通常被封装在路由拦截器或中间件中）：

1. **前端发起请求：** 携带凭证（例如 JWT Token）。
2. **认证中间件 (AuthN Middleware) 拦截：**
* 校验 Token 是否合法、是否过期。
* 如果非法，直接打回 `401 Unauthorized`。
* 如果合法，解析出 `UserID`，将其注入到当前请求的上下文（Context）中。


3. **授权中间件 (AuthZ Middleware) 拦截：**
* 从上下文中拿出 `UserID`，获取当前请求的 URL（如 `/api/data`）和 Method（如 `DELETE`）。
* 调用权限引擎（如 `Enforcer.Enforce(UserID, "/api/data", "DELETE")`）。
* 如果返回 false，打回 `403 Forbidden`。
* 如果返回 true，放行。


4. **到达业务逻辑层：** 此时控制器（Controller）可以绝对信任该用户既合法又有权限，专注于处理业务。

## Casbin 极简核心总结笔记

### 1. 核心定位
* **做的是“授权 (AuthZ)”**：解决“这个人能不能做这件事”（决定放行还是 403 Forbidden）。
* **不做“认证 (AuthN)”**：不管登录、不管密码校验、不管 JWT 解析（这是前置中间件做的事）。
* **最大价值**：将权限判断从业务代码中抽离，代码里只需要写一句 `Enforce()`，不需要写满屏幕的 `if role == "admin"`。

### 2. 核心架构（PERM 元模型）
所有权限体系都被浓缩在一个 `.conf` 配置文件中：

* **R (Request - 请求)**
    * `r = sub, obj, act`
    * 白话：实际发生的动作（比如：张三(sub) 访问 机房(obj) 执行 开门(act)）。
* **P (Policy - 策略)**
    * `p = sub, obj, act`
    * 白话：数据库里的白名单规则（比如：管理员(sub) 对 机房(obj) 有 开门(act) 权限）。
* **E (Effect - 结果)**
    * `e = some(where (p.eft == allow))`
    * 白话：白名单机制。查遍所有策略，只要有一条亮绿灯，就放行。
* **M (Matcher - 匹配器)**
    * `m = r.sub == p.sub && r.obj == p.obj && r.act == p.act`
    * 白话：权限引擎的大脑。定义请求(r) 和 策略(p) 具体怎么核对。

### 3. 进阶核心：分组机制 (Role Definition)
支持在多个维度上打包归类（即 RBAC 模型的基础）：
* `g = _, _` (人员分组)：比如 `g(张三, 管理员)` -> 把张三塞进管理员组。
* `g2 = _, _` (资源分组)：比如 `g2(/api/user/1, API组A)` -> 把具体接口塞进资源组。
* `g3 = _, _` (动作分组)：比如 `g3(GET, 浏览)` -> 把 HTTP 动作塞进权限动作组。

### 4. 经典实战 Matcher 解析
配置：`m = g(r.sub, p.sub) && keyMatch2(r.obj, p.obj) && r.act == p.act`
翻译为三大通关条件：
1. **身份关 `g(r.sub, p.sub)`**：你是这个角色吗？（支持向上追溯角色组）。
2. **资源关 `keyMatch2(...)`**：你访问的 URL 和规则里的 RESTful 路径（如 `/api/:id`）能对上吗？
3. **动作关 `r.act == p.act`**：你做的操作（如 POST）是被允许的吗？
（三个 `&&` 全为 True 才算匹配成功）。

### 5. 代码层面的工作流
1.  系统启动：`Enforcer` 通过 `Adapter` 把数据库里的所有策略(Policy)加载到内存。
2.  请求到达：前置拦截器查明“你是谁”（如解析 JWT 拿到 UserID）。
3.  权限校验：调用 `Enforcer.Enforce(UserID, 请求路径, HTTP方法)`。
4.  执行结果：返回 `true` (进入业务代码) 或 `false` (返回 403 滚蛋)。
