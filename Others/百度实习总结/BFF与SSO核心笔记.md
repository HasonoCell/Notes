# `server` BFF 与 SSO 核心笔记

这份笔记不再按源码文件逐个解释，而是把注意力放在两个核心问题上：

1. 这个 `server` 作为 **BFF / 接入层**，到底解决了什么问题？
2. 它接入 **UUAP 单点登录（SSO）** 的思路是什么？

目标不是“记住代码细节”，而是先建立一套更容易复用、内化的理解框架。

---

## 1. 先用一句话定义这个 `server`

可以把它理解成：

> **位于前端页面和真正业务后端之间的一层 Node BFF，同时负责接入企业统一认证。**

它本身不是主要业务服务，而是一个“承上启下”的中间层。

它向上面对前端，向下连接两类后端能力：

- **认证中心 UUAP**：解决“你是谁、你是否已登录”
- **业务服务**：解决“你要查什么、提交什么、生成什么”

所以它更像一个“总入口”，而不是“最终业务处理者”。

---

## 2. 什么是 BFF，为什么这里需要它

### 2.1 BFF 的本质

BFF（Backend For Frontend）可以直译为“为前端服务的后端”。

它的核心思想不是去承载所有业务，而是：

- **替前端屏蔽复杂后端细节**
- **把多个后端能力整合成前端更容易使用的接口**
- **承担前端不适合直接做的事情**，比如认证、签名、转发、cookie 管理、权限前置校验

### 2.2 为什么这个项目需要 BFF

如果没有这层 BFF，前端会直接面对很多复杂问题：

- 该向哪个后端地址发请求
- 哪些请求要先校验登录
- 如何和 UUAP 打交道
- 如何处理 `pToken` / `sToken`
- 如何构造登录跳转和回调
- 如何在请求里补充当前用户身份
- 如何统一处理跨域、错误、转发等问题

把这些逻辑都放在前端会带来几个问题：

1. **前端复杂度过高**
2. **安全边界不清晰**
3. **认证细节暴露过多**
4. **接口适配逻辑分散，后续难维护**

所以这个 `server` 的价值就在于：

> **把“认证接入 + 请求整理 + 后端转发”统一收口到服务端。**

---

## 3. 用职责而不是文件来理解这层 BFF

理解这层 BFF，最好的方式不是先背文件名，而是先抓住它承担的 4 类职责。

### 3.1 职责一：作为统一入口

对前端来说，这一层就是统一入口：

- 页面资源从这里出去
- API 请求从这里进去
- 登录态校验从这里开始

也就是说，前端面对的不是一堆分散的服务，而是这个 BFF。

### 3.2 职责二：作为认证接入层

它负责把系统接入 UUAP，并把认证流程“翻译”成当前系统可用的登录态。

这层做的不是“用户系统”本身，而是“把统一认证的结果接进来”。

简单说就是：

- 没登录时，把你送去 UUAP
- 登录完成后，把 UUAP 返回的凭证换成本系统可用的 token
- 后续每次请求再帮你校验登录态

### 3.3 职责三：作为请求转发层

它把前端发来的请求转发给真正的业务后端。

这里的“转发”不是机械地透传，而是会做一些加工：

- 补充当前用户身份
- 补充额外字段
- 把前端接口路径映射到下游服务路径
- 统一错误处理和返回格式

所以它像网关，但又不是纯网关。

### 3.4 职责四：作为边界整理层

BFF 很重要的一个作用是“边界整理”：

- 前端只关心页面交互与调用统一接口
- 业务后端只关心业务数据与业务处理
- 认证中心只关心身份认证
- BFF 负责把这三者接起来

这会让整个系统分层更清楚。

## 4. 这个 `server` 在系统里的位置

可以先建立这样一个心智模型：

```text
浏览器 / 前端页面
        │
        ▼
   Node BFF（当前 server）
      ├─ 对接 UUAP
      └─ 转发到业务后端
```

如果把它再展开一点：

```text
前端
  │
  ├─ 访问页面资源
  └─ 调用统一 API
          │
          ▼
      BFF / server
          │
          ├─ 检查登录态
          ├─ 必要时跳转 UUAP
          ├─ 把认证结果转换为系统自己的可用状态
          └─ 把业务请求转发到真正后端
```

从架构角色上看，它既像：

- **应用网关**：统一接流量、做前置处理
- **BFF**：为前端做协议适配、请求整形、认证承接

但它不是：

- 核心业务引擎
- 统一认证中心本身
- 纯静态资源服务器
- 纯 NGINX 反向代理

---

## 5. 为什么说它像网关，但不等于 NGINX

这个类比是对的，但要加一个限制：

> **它具备网关作用，但它是“带应用逻辑的网关”，不是“纯流量转发器”。**

### 5.1 它像 NGINX 的地方

- 都是统一入口
- 都可以做转发
- 都可以挂在前端和后端之间
- 都可以承担一部分跨域、静态资源、请求分发职责

### 5.2 它不像 NGINX 的地方

它额外做了很多“只有应用层代码才能方便完成”的事：

- 读写 cookie
- 和 UUAP 做服务端认证交互
- 构造登录回调逻辑
- 判断哪些接口需要登录，哪些不需要
- 改造请求体，补充用户身份
- 做定制化错误处理

所以更准确的判断是：

> **NGINX 更像“通用入口设施”，而这个 `server` 更像“带认证和请求加工能力的应用层入口”。**

这个区别非常重要，因为它决定了：

- NGINX 侧重基础流量治理
- BFF 侧重面向业务与前端的适配

---

## 6. 理解这层 BFF 的关键：它主要不生产业务，它主要“组织业务访问”

很多人第一次看这类代码会困惑，是因为会下意识把它当成“业务后端”来看。

但其实它的主要任务不是：

- 算法处理
- 核心业务决策
- 数据存储
- 复杂领域逻辑

它更像在做下面这些事：

- 用户有没有资格继续访问
- 当前请求该不该放行
- 当前请求该转发到哪里
- 转发前要不要补充用户信息
- 如果没登录，该怎么把用户引到认证中心
- 登录回来后，如何把认证状态接上

所以理解它最好的关键词不是“业务逻辑”，而是：

- **接入**
- **认证**
- **转发**
- **整形**
- **边界**

---

## 7. 从用户视角理解：一次请求究竟发生了什么

不要先从函数名出发，先从用户视角出发。

一个用户访问系统时，大致会经历两类链路：

### 7.1 链路一：访问页面 / 访问接口时发现未登录

发生的事情是：

1. 用户访问系统
2. BFF 先检查当前请求是否带了系统需要的登录凭证
3. 如果没有，就不让它直接访问业务后端
4. BFF 把用户引导到 UUAP 去登录
5. UUAP 登录完成后，再把用户带回系统
6. 系统把 UUAP 返回的结果转换为本系统能识别的登录态
7. 用户随后就能正常访问业务接口

这是一条“先认证，再访问业务”的链路。

### 7.2 链路二：用户已经登录，正常访问业务接口

发生的事情是：

1. 用户发起业务请求
2. BFF 检查登录态是否仍然有效
3. 如果有效，提取当前用户身份
4. BFF 把用户身份拼进请求中
5. 再把请求转发给真正的业务后端
6. 后端返回结果，BFF 回给前端

这是一条“认证通过后再做代理转发”的链路。

## 8. SSO 到底在解决什么问题

单点登录（SSO）的核心价值不是“少输一次密码”这么简单，它真正解决的是：

> **多个内部系统如何共享同一套身份认证结果。**

对企业内部系统来说，理想状态是：

- 用户只需要在统一认证平台登录一次
- 之后访问系统 A、系统 B、系统 C 时，都不需要重复登录
- 每个系统仍然可以保有自己的访问控制和服务边界

这就要求有一个统一身份来源，这里就是 UUAP。

所以从系统分工上看：

- **UUAP 负责认证“你是谁”**
- **系统 A 负责决定“你能在我这里做什么”**

这个分工非常关键。

## 9. SSO 这套机制里的几个关键概念

为了真正理解这层登录体系，需要先把几个名词分清楚。

### 9.1 `pToken`

可以把它理解成：

> **平台级登录态凭证**

它偏向说明：你在统一认证平台这一侧是否已经完成登录。

它更多体现的是“你在统一认证中心是不是已登录用户”。

### 9.2 `sToken`

可以把它理解成：

> **某个具体系统可用的服务级凭证**

它说明的不是“你在整个认证平台登录了没有”，而是：

- 统一认证中心已经确认过你的身份
- 当前这个业务系统可以接受这个结果
- 后续你可以带着它继续访问这个系统

### 9.3 `ticket`

可以把它理解成：

> **一次性的兑换凭证**

它不是长期登录态，而是登录完成后的一次性中转票据。

它的价值在于：

- 避免直接把最终服务凭证暴露在登录跳转链路里
- 让系统 A 自己去服务端向 UUAP 换取真正可用的 `sToken`
- 通过“一次性、短时有效”的设计降低重放风险

### 9.4 `appKey`

它代表的是：

> **当前来发起认证的系统身份**

认证中心需要知道“是谁在请求认证”，因此系统 A 要带着自己的身份去发起登录。

### 9.5 `service`

它代表的是：

> **登录完成后应该回到哪里**

也就是登录成功后的回调地址。

## 10. 把标准 SSO 链路讲清楚

下面不先对照代码，先用架构思路把这条链路讲顺。

### 10.1 阶段一：用户访问业务系统

用户先访问系统 A。

这时系统 A 不会轻易默认“你已经登录”，而是要先检查：

- 你有没有系统侧可用的登录态
- 你是不是已经通过统一认证

如果缺少必要凭证，系统 A 会认为当前请求还不可信。

这体现的是一种典型思路：

> **先验证身份，再放行业务访问。**

### 10.2 阶段二：系统 A 把用户送去认证中心

当系统 A 判断当前用户尚未完成系统侧登录时，它会把用户重定向到 UUAP。

重定向时，系统 A 至少要告诉 UUAP 两件事：

- 我是谁：`appKey`
- 你登录完成后要回哪里：`service`

这一步相当于说：

> “我是系统 A，这个用户想访问我。请你先帮我确认身份，确认好了再把他送回来。”

### 10.3 阶段三：用户在 UUAP 完成统一认证

用户在 UUAP 上完成登录。

此时 UUAP 会建立自己的平台级登录态，并且生成一个一次性 `ticket`。

这个阶段的关键点是：

- 平台确认了用户身份
- 但系统 A 还没有直接拿到自己的服务态凭证
- 所以还需要一个“回跳 + 兑换”的过程

### 10.4 阶段四：UUAP 带着 `ticket` 把用户送回系统 A

UUAP 根据之前收到的 `service`，把浏览器重定向回系统 A。

这个回来的请求里会带着 `ticket`。

注意，这时系统 A 仍然不会简单相信浏览器说“我登录过了”，而是会继续做下一步：

> **由系统 A 自己作为服务端，拿 `ticket` 去向 UUAP 再确认一次。**

这就是“零信任”的体现：

- 浏览器带回来的只是中转凭证
- 是否真的认可，还要由系统 A 自己向认证中心核验

### 10.5 阶段五：系统 A 用 `ticket` 向 UUAP 换取 `sToken`

系统 A 在服务端发起请求，把 `ticket` 交给 UUAP。

如果 UUAP 校验通过，就返回系统 A 可使用的 `sToken`。

这一阶段的设计很重要，因为它带来了几个好处：

1. **服务凭证不是直接经由前端“生成”的**
2. **系统 A 自己参与了最后一步身份确认**
3. **ticket 一次性使用，可以降低被重放的风险**

### 10.6 阶段六：系统 A 建立自己的登录态

系统 A 拿到 `sToken` 以后，就可以把它保存为当前系统可识别的登录态。

此后用户再访问系统 A，就会自动带上系统 A 所需的 token。

也就是说：

- `UUAP` 解决了“统一认证”
- `系统 A` 解决了“本系统会话建立”

这是两个层次，不要混在一起。

### 10.7 阶段七：后续请求继续校验，再访问业务

后续每次业务请求来到系统 A：

1. 系统 A 先检查凭证
2. 再确认登录态是否有效
3. 校验通过后，提取当前用户身份
4. 把请求转发给业务后端

也就是说，SSO 不是“登录一次后系统永远不再校验”，而是：

> **先统一登录，再按系统自己的规则持续校验和消费这个登录态。**

## 11. 为什么这套 SSO 设计值得记住

这套流程背后，其实有几条非常重要的架构原则。

### 11.1 认证中心和业务系统解耦

UUAP 不负责你的业务逻辑，业务系统也不负责统一认证本身。

两者的分工是清晰的：

- UUAP：统一身份认证
- 系统 A：本系统接入、会话消费、业务访问控制

### 11.2 浏览器不是最终可信方

浏览器带回来的 `ticket` 不能直接等同于“系统登录成功”。

系统 A 还要再自己和 UUAP 通信，确认票据有效，才能建立系统自己的登录态。

这能有效降低伪造和重放风险。

### 11.3 平台登录态和系统登录态分层

`pToken` 和 `sToken` 分层的意义很大：

- 一个偏“统一认证平台层面”
- 一个偏“具体业务系统层面”

这种分层能兼顾：

- 统一登录体验
- 具体系统的独立边界

### 11.4 BFF 是认证与业务之间的缓冲带

如果没有 BFF，这些事情就会变得很别扭：

- 前端直接理解认证细节，负担太重
- 业务后端直接兼容页面访问，边界会混乱
- 统一认证结果很难自然地接到前端调用链里

BFF 的存在，就是为了把这些复杂度收口。

## 12. 从“思路”角度理解这层 BFF 的工作流

把细节抽掉以后，这层 BFF 的工作流其实很简洁：

### 12.1 入口判断

先判断当前请求属于哪一类：

- 静态资源请求
- 免登录请求
- 需要登录的业务请求

### 12.2 认证前置

对于需要登录的请求：

- 检查 token
- 必要时跳转统一认证
- 回调后建立本系统登录态

### 12.3 身份注入

在确认当前用户是谁之后：

- 把用户身份挂到上下文
- 或拼进下游请求参数中

### 12.4 业务转发

最后才是：

- 把请求路由到真正业务后端
- 把后端结果返回给前端

也就是说，它的运行顺序本质上是：

> **判断请求 → 校验身份 → 建立上下文 → 转发业务**

这 4 步就是这类 BFF 的通用骨架。

## 13. 如何更容易地内化这套知识

如果你想把这部分知识真正变成自己的，而不是只记住一堆名词，建议记住下面这几个问题。

### 13.1 第一个问题：这层服务到底替谁服务？

答案是：

- 直接服务前端
- 间接连接认证中心和业务后端

所以它不是“业务本身”，而是“访问业务的组织者”。

### 13.2 第二个问题：这层服务最核心的价值是什么？

答案是两点：

- **承接认证**
- **整理访问**

### 13.3 第三个问题：为什么不让前端直接调 UUAP 和业务后端？

因为那会让前端承担过多：

- 安全敏感逻辑
- token 细节
- 登录跳转拼装
- 多后端适配
- 用户身份注入

这些更适合收口到服务端。

### 13.4 第四个问题：为什么不直接用 NGINX 替代？

因为 NGINX 更适合做基础流量治理，而这里还需要：

- 应用层认证接入
- 请求体加工
- 业务字段补充
- 针对具体接口的定制逻辑

这已经超出了纯反向代理的舒适区。

### 13.5 第五个问题：这套设计里最值得记住的架构图是什么？

就是下面这张简化图：

```text
前端
  │
  ▼
BFF / server
  ├─ 先处理认证
  ├─ 再建立用户上下文
  └─ 最后转发到业务后端

认证能力来自 UUAP
业务能力来自真正后端
```

把这张图真正记住，后面的代码细节都会更容易理解。

## 14. 这份笔记最后想帮你建立的三个结论

### 结论一：`server` 的本质不是“业务服务”，而是“接入层”

它更关心：

- 入口
- 认证
- 转发
- 整形

而不是核心业务计算。

### 结论二：BFF 的核心价值是“收口复杂度”

它把前端原本不适合承担的事情统一放在服务端处理：

- 登录接入
- token 管理
- 请求身份补充
- 后端适配

### 结论三：SSO 的核心思路是“统一认证 + 系统自建会话”

不是说用户在 UUAP 登录完，一切就自动结束了。

真正完整的过程是：

1. 统一认证平台先确认身份
2. 业务系统再把这个结果转换成自己可消费的登录态
3. 之后业务访问仍然由业务系统自己控制

这才是 SSO 真正落到系统架构里的方式。

## 15. 一页速记版

如果最后只保留最核心的记忆，可以记这一页：

### `server` 是什么

- 是前端前面的 Node BFF / 应用接入层
- 不负责核心业务计算
- 负责认证接入、请求整理、接口转发

### 它解决什么问题

- 屏蔽前端对多后端和认证中心的直接复杂交互
- 统一登录态校验
- 把用户身份注入业务请求
- 把前端接口映射到真正业务后端

### 它和 NGINX 的关系

- 有网关作用
- 但不是纯反向代理
- 它带有应用层认证和请求加工逻辑

### SSO 的主线

- 未登录 → 跳 UUAP
- UUAP 认证成功 → 返回一次性 `ticket`
- 系统 A 用 `ticket` 换 `sToken`
- 系统 A 建立自己的登录态
- 后续请求继续校验，再访问业务

### 真正要记住的架构思想

- **统一认证不等于系统会话本身**
- **浏览器带回票据，不代表系统直接信任**
- **BFF 是认证链路和业务链路之间的衔接层**

---

## 16. 结束语

如果你以后再看类似代码，不要先问“这个函数在干嘛”，而要先问：

1. 这一层在整个系统里扮演什么角色？
2. 它是在解决认证问题，还是在解决业务问题？
3. 它是在生产业务，还是在组织业务访问？

一旦先把这三个问题想清楚，这类 BFF + SSO 的代码就会一下子顺很多。
