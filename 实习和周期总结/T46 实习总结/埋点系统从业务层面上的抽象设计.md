# 如何设计一个埋点系统？从 Session、Flow 与 Event 三个业务抽象去思考

## 1. 核心结论

从业务层面看，一个埋点系统首先要回答三个问题：

1. 用户做了什么？
2. 这些动作是不是同一件事情？
3. 这件事情最后成功还是失败？

这三个问题分别对应三个层次：

```text
Session：用户这次访问 App 的整体经历
  ↓
Flow：用户正在完成的一件具体事情
  ↓
Event：这件事情中的一个具体动作
```

## 2. 三个 ID 的业务含义

### 2.1 `session_id`：一次 App 访问

`session_id` 表示用户一次使用 App 的经历。

例如用户打开 App 后，可能依次浏览内容、填写资料、查看结果和发起分享，这些行为可以属于同一个 App session。

一次 session 可以包含多个业务流程：

```text
session_id = S1
│
├── flow_id = F1：一次核心业务流程
└── flow_id = F2：一次互动流程
```

`session_id` 解决的是：

```text
用户是在什么访问背景下完成这些行为的？
```

它不是某一项具体业务的编号。

### 2.2 `flow_id`：一件完整的业务事情

`flow_id` 表示用户正在完成的一件具体事情。

例如：

```text
一次注册流程
一次交易流程
一次售后流程
一次内容消费流程
一次分享或邀请流程
```

一个 flow 应该明确：

```text
什么时候开始？
有哪些步骤？
什么结果算成功？
什么结果算失败、取消、放弃或过期？
```

`flow_id` 解决的是：

```text
这些离散动作是不是属于同一件业务？
```

### 2.3 `event_id`：一次具体动作

`event_id` 是一条事件的唯一编号，表示某一次具体发生的动作。

事件类型由 `event_name` 表示，例如：

```text
event_name = flow_step_completed
event_id = E10086
```

一个 flow 可以包含多个 event：

```text
flow_id = F1
│
├── event_id = E1：进入流程
├── event_id = E2：完成第一个步骤
├── event_id = E3：提交业务操作
└── event_id = E4：流程成功
```

`event_id` 解决的是：

```text
这一条具体动作是否已经被记录过？
```

## 3. 三层关系示例

一个用户在一次 App 使用过程中完成两个不同流程：

```text
session_id = S1
│
├── flow_id = F1：核心业务流程
│   ├── event_id = E1：进入流程
│   ├── event_id = E2：完成第一个步骤
│   ├── event_id = E3：提交业务操作
│   ├── event_id = E4：第一次尝试失败
│   ├── event_id = E5：再次执行操作
│   └── event_id = E6：第二次尝试成功
│
└── flow_id = F2：互动流程
    ├── event_id = E7：发起互动
    └── event_id = E8：互动成功
```

因此，三者关系是：

```text
一个 session 可以包含多个 flow
一个 flow 可以包含多个 event
一个 event 属于某个 flow，并发生在某个 session 中
```

在数据上通常表现为：每条 event 同时带有相关的 `session_id` 和 `flow_id`。

## 4. 为什么不能只使用 `session_id`

假设用户在一次 App 访问中完成：

```text
查看对象 A
完成对象 A 相关流程
发起互动
查看对象 B
完成对象 B 相关流程
```

如果只有 `session_id`，我们只能知道这些事件都发生在同一次 App 访问中，却无法判断：

```text
哪些事件属于对象 A 的业务流程？
哪些事件属于对象 B 的业务流程？
哪些事件属于互动流程？
```

因此：

```text
session_id 是访问背景
flow_id 是业务边界
event_id 是动作明细
```

## 5. 业务流程 ID 的通用设计

当多个离散事件共同组成一件业务时，应为这件业务生成一个流程 ID，例如 `flow_id`。

它的流程大致是：

```text
flow_id = F1
│
├── flow_exposure：用户进入流程
├── flow_step：完成一个步骤
├── flow_step：改变或确认选择
└── flow_outcome：流程最终结果
```

这个设计的本质是：

```text
给一次业务流程生成一个唯一编号
让该流程中的多个事件共享这个编号
保存用户在流程中的步骤和状态
记录最终结果
```

它的优点包括：

- 流程开始、步骤和结果可以归属于同一件业务；
- 没有业务意义的重复操作可以被忽略；
- 有业务意义的选择变化和重试可以被记录；
- 最终结果可以保证只记录一次；
- 流程跨页面或跨组件后仍能继续关联；
- 状态逻辑可以单独测试。

## 6. 不能把所有重复事件都删除

重复事件需要先判断它是不是有业务意义。

### 没有业务意义的重复

```text
选择 A → 选择 A → 选择 A
```

后两次可能只是重复点击，可以抑制。

### 有业务意义的重复

```text
选择 A → 选择 B
```

这是用户真实改变了选择，应当保留，并记录：

```text
previous_choice
choice
is_switch
choice_index
switch_count
```

同样地，用户多次进入某个步骤或重复提交，不一定是错误数据，可能表示流程中的真实重试。正确做法不是简单删除，而是记录：

```text
flow_id
page_view_id
attempt_id
reentry_index
```

这样可以区分：

```text
一次流程中的多次尝试
```

和：

```text
多次完全独立的流程
```

## 7. 流程需要定义结果

一个 flow 不一定只有成功和失败两种结果，还可能有：

```text
success       成功
failed        失败
cancelled     用户主动取消
abandoned     用户中途离开
expired       超时失效
processing    处理中
```

例如一个需要提交和确认的业务流程可以定义为：

```text
flow_started
step_completed
operation_failed
operation_retried
flow_succeeded
```

只有定义了最终结果，产品和增长人员才能计算真正的转化率和流失率。

## 8. 当前阶段的设计原则

后续设计埋点时，建议按照以下顺序思考：

1. 先定义一件业务事情是什么。
2. 定义这件事情什么时候开始。
3. 列出中间步骤。
4. 定义成功、失败、取消、放弃和过期状态。
5. 为这件事情生成 `flow_id`。
6. 再为每个具体动作设计事件，也就是 `event`。
7. 最后将事件与用户、设备和 App session 关联起来。

## 9. 最终结论

埋点系统的核心不是收集尽可能多的点击，而是建立一套能解释用户行为的层次结构：

```text
Session：用户这次访问 App 的背景
Flow：用户正在完成的一件完整业务
Event：业务流程中的具体动作
```

一个好的业务流程 ID 设计，应当把多个离散事件收敛成一次可分析的完整业务流程。

后续可以将这种模式应用到注册、交易、内容消费、售后、分享和其他需要多步完成的业务中。
