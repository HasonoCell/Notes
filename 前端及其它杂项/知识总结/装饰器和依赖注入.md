装饰器是 Python 中非常强大且优雅的语法糖，它允许我们在不修改原函数代码的情况下，动态地增加功能。

装饰器可以分三个阶段来学习：

1.  **基础概念**：什么是装饰器，它是如何工作的。
2.  **进阶用法**：带参数的装饰器、类装饰器、`functools.wraps`。
3.  **实战应用**：使用装饰器实现一个简洁的“依赖注入”系统。

---

### 什么是装饰器

在 Python 中，**函数是一等公民**。这意味着函数可以像变量一样被传递、作为参数、作为返回值。装饰器本质上就是一个**返回函数的函数**。

#### 1. 最简单的装饰器

假设我们有一个简单的函数 `hello`，我们想在它执行前后打印日志，但不想修改 `hello` 的内部代码。

```python
def my_logger(func):
    def wrapper():
        print(">>> Log: 函数开始执行")
        func()  # 执行原始函数
        print(">>> Log: 函数执行结束")
    return wrapper

@my_logger
def hello():
    print("Hello, World!")

# 调用
hello()
```

**原理解析：**
`@my_logger` 等价于执行了 `hello = my_logger(hello)`。
当你调用 `hello()` 时，实际上是在调用 `wrapper()`，而在 `wrapper()` 中，又再去调用被装饰的函数。

#### 2. 处理参数和返回值

如果原函数有参数和返回值，我们的 `wrapper` 也必须能够处理它们。通常使用 `*args` 和 `**kwargs`。

```python
def timer(func):
    import time
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)  # 接收返回值
        end = time.time()
        print(f"函数 {func.__name__} 耗时: {end - start:.4f} 秒")
        return result  # 返回原函数的返回值
    return wrapper

@timer
def add(a, b):
    return a + b

print(f"结果: {add(10, 20)}")
```

### 装饰器的进阶用法

#### 1. 保留原函数的元数据 (`functools.wraps`)

使用装饰器后，原函数的 `__name__`（函数名） 和 `__doc__`（函数文档注释） 会变成 `wrapper` 的。为了避免这个问题，我们需要使用标准库的 `functools.wraps`。

```python
import functools

def my_decorator(func):
    @functools.wraps(func)  # 关键行：保留原函数的元数据
    def wrapper(*args, **kwargs):
        """这是 wrapper 的文档"""
        return func(*args, **kwargs)
    return wrapper

@my_decorator
def example():
    """这是原函数的文档"""
    pass

print(example.__name__)  # 输出: example (如果没有 @wraps，会输出 wrapper)
print(example.__doc__)   # 输出: 这是原函数的文档（如果没有 @wraps，会输出"这是 wrapper 的文档"）
```

#### 2. 带参数的装饰器（三层嵌套）

如果你想给装饰器本身传参（比如设置重试次数、日志级别），你需要再包裹一层函数。

```python
def repeat(num_times):
    def decorator_repeat(func):
        @functools.wraps(func)
        def wrapper(*args, **kwargs):
            for _ in range(num_times):
                result = func(*args, **kwargs)
            return result
        return wrapper
    return decorator_repeat

@repeat(num_times=3)
def greet(name):
    print(f"Hello {name}")

greet("World")
```

### 实现基于装饰器的依赖注入（DI）

依赖注入（Dependency Injection, DI）是一种设计模式，用于解耦组件。通过装饰器，我们可以利用 Python 的 **类型注解 (Type Hints)** 来实现一个非常 Pythonic 的自动注入系统。

```python
import functools
from typing import Any, Dict, Type


class Container:
    """简单的依赖注入容器"""

    _services: Dict[Type, Any] = {} # 定义依赖注入容器的服务注册列表

    @classmethod # @classmethod 装饰器，用来创建一个静态方法
    def register(cls, interface: Type, implementation: Any):
        """注册依赖"""
        cls._services[interface] = implementation
        print(f"[Container] 注册服务: {interface.__name__} -> {implementation}")

    @classmethod
    def resolve(cls, interface: Type) -> Any:
        """获取依赖"""
        return cls._services.get(interface)

def Service(cls):
    """
    类装饰器：自动将类实例化并注册到 DI 容器中。
    """
    instance = cls()
    Container.register(cls, instance)
    return cls # 返回这个类本身，保持类定义不变

def Inject(func):
    """
    依赖注入装饰器，它会检查函数的类型注解。
    如果函数的参数（即函数所需要的依赖）的类型在容器中注册过，
    就自动注入实例。
    """

    @functools.wraps(func) # 保留被装饰函数的名称和文档注释
    def wrapper(*args, **kwargs):
        # 获取被装饰函数的类型提示
        hints = func.__annotations__

        # 遍历参数，查看是否需要注入
        # 注意：这里是一个简化的实现，主要演示原理
        new_kwargs = kwargs.copy()

        # 获取函数的所有参数名
        code_obj = func.__code__
        arg_names = code_obj.co_varnames[: code_obj.co_argcount]

        for arg_name in arg_names:
            # 如果参数已经通过 args 或 kwargs 传递了，则跳过
            if arg_name in new_kwargs:
                continue

            # 检查该参数是否有类型注解
            arg_type = hints.get(arg_name)
            if arg_type:
                # 尝试从容器中解析依赖
                dependency = Container.resolve(arg_type)
                if dependency:
                    new_kwargs[arg_name] = dependency
                    print(f"[Injector] 自动注入 {arg_name}: {dependency}")

        return func(*args, **new_kwargs)

    return wrapper


# --- 定义一些服务，将其注册到容器  ---

@Service
class Database:
    def query(self, sql):
        return f"执行查询: {sql}"

@Service
class EmailService:
    def send(self, content):
        return f"发送邮件: {content}"

# --- 业务逻辑 ---

@Inject
def create_user(
    username: str, db: Database = None, email: EmailService = None
):
    """
    注意：调用此函数时，不需要传递 db 和 email，
    装饰器会根据类型注解自动注入！

    (这里给 db 和 email 设置默认值 None 是为了骗过 Pylance 等静态检查工具，
    避免它报错说我们调用时少传了参数。实际上装饰器会在运行时填入正确的值)
    """
    print(f"\n正在创建用户: {username}")

    if db and email:
        print(db.query(f"INSERT INTO users VALUES ({username})"))
        print(email.send(f"欢迎 {username}！"))
    print("用户创建成功。")


# --- 运行测试 ---

if __name__ == "__main__":
    # 我们只需要传递 username，其他依赖被自动注入了
    create_user(username="hasono")

    # 输出：
    # [Container] 注册服务: Database -> <__main__.Database object at 0x10272be90>
    # [Container] 注册服务: EmailService -> <__main__.EmailService object at 0x10272bef0>
    # [Injector] 自动注入 db: <__main__.Database object at 0x10272be90>
    # [Injector] 自动注入 email: <__main__.EmailService object at 0x10272bef0>

    # 正在创建用户: hasono
    # 执行查询: INSERT INTO users VALUES (hasono)
    # 发送邮件: 欢迎 hasono！
    # 用户创建成功。

```

#### 代码解析

1.  **`Container` 类**：这是一个简单的注册表，或者说 DI 容器，用于存储“接口（类型）”到“具体实例”的映射。

2.  **`@Service` 装饰器**：注册服务。

    -   它会把类实例化，并注册到 DI 容器中。
    -   简化了实例的手动创建和注册，避免了手动调用 `Container.register`。
    -   也被叫做“声明式注册”。

3.  **`@Inject` 装饰器**：这是核心。

    -   它拦截函数的调用。
    -   它读取函数的 `__annotations__`（类型注解）。
    -   它检查参数是否在 `Container` 中注册过。
    -   如果注册过，它就自动从容器中取出实例，并通过 `kwargs` 注入给函数。

4.  **业务逻辑**：`create_user` 函数只需要声明它需要 `Database` 和 `EmailService` 类型，装饰器会自动填入这些参数，调用者完全不需要关心依赖的创建。

#### 装饰器与依赖注入关系的进一步理解

依赖注入其实可以分开来理解，即“依赖”和“注入”，举个例子，比如 create_user 这个函数（即“**高层模块**”），它的”依赖“就是数据库和邮箱（即“**底层服务**”），因为新建一个 user 需要注册邮箱，同时需要将创建出来的 user 写入到数据库。

现在有两种方式可以让 create_user 这个函数使用这两个依赖，一种方式是当每次函数调用时在函数内部创建依赖的实例（即“**硬编码依赖**”），这种方法不好，因为它使得函数和自身所需的依赖严重耦合，且会造成依赖的多个实例化。此外，如果我想将 Database 这个底层服务换为 MySQLDatabase 这个服务，就需要手动修改 cerate_user 的源代码，违反了“**开闭原则**”。

另一种方式是，提前在全局创建好依赖的实例，并在 create_user 被调用时通过装饰器拦截其调用，然后依靠装饰器将创建好的实例自动”注入“到函数当中，函数则通过参数的形式来接收依赖实例。此方法是的函数与所需的依赖解耦，并且避免了重复实例化。这种操作也叫“**控制反转（Inversion of Control，IoC）**”，即控制权发生了转移，以前是 create_user 自己控制如何创建依赖，现在是外部 DI 容器控制依赖创建过程，create_user 只是被动接收。

**装饰器最大的魅力在于：**

业务函数的视角中，create_user 只想关注“如何创建用户”，它不想关心“数据库连接是怎么来的”。

装饰器的视角中，@inject 只关注“如何组装参数”，它不关心“用户是怎么创建的”。

通过装饰器，我们将“**业务逻辑”（Business Logic**） 和 “**基础设施逻辑”（Infrastructure Logic/Wiring**）完全剥离了。代码看起来非常干净，create_user 里面没有一行代码是在做 Container.resolve() 这种杂事。

Python 的装饰器之所以能把依赖注入做得这么优雅（比 Java 或 C++ 的某些实现更简洁），是因为它结合了 装饰器 + 类型提示（Type Hints）。装饰器可以拿到被装饰函数的对象。通过 **func.annotations**，装饰器可以“偷看”函数需要什么类型的参数。通过 **kwargs**，装饰器可以动态地修改传递给函数的参数。
