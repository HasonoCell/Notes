我们设定一个统一的“购买商品”场景：用户 U 购买 1 件商品 A，单价 100 元。整个分布式事务需要跨越三个微服务，必须全部成功或全部失败：

1. **订单服务 (Order)：** 创建一笔状态为“已支付”的订单。
2. **库存服务 (Inventory)：** 商品 A 的库存扣减 1。
3. **资金服务 (Account)：** 用户 U 的账户余额扣减 100 元。

# 2PA 

核心思想是悲观锁，强一致性。协调者掌控全局，所有数据库资源被锁死，直到最后一刻才同时释放。通常依赖底层数据库的 XA 协议支持。2PA 基本依赖数据库的本地事务，比如 MySQL，如果有任何一个服务的本地事务失败，那么回滚都是依靠 MySQL 自己的 undo log 实现，这也是性能瓶颈的原因，也是下一种方案 TCC 和 2PA 最大的不同。

**执行流程：**

* **阶段一 (Prepare)：** 协调者向订单、库存、资金三个数据库发送 `PREPARE` 指令。三个数据库各自在本地开启事务，执行 SQL（写入记录、扣库存、扣钱），**但不执行 COMMIT**。此时，对应的数据行被加上排他锁。
* **阶段二 (Commit/Rollback)：** 如果三个数据库都回复“准备就绪”，协调者发送 `COMMIT`，三个数据库同时提交，释放行锁。如果任何一个超时或报错，协调者发送 `ROLLBACK`，大家一起回滚，释放行锁。

```go
// 伪代码：2PC 协调者逻辑
func Purchase2PC() error {
    // 阶段一：开启所有参与者的本地事务，并执行操作（但不提交）
    err1 := dbOrder.Prepare("INSERT INTO orders (user, item, status) VALUES ('U', 'A', 'PAID')")
    err2 := dbInventory.Prepare("UPDATE inv SET stock = stock - 1 WHERE item = 'A'")
    err3 := dbAccount.Prepare("UPDATE acc SET balance = balance - 100 WHERE user = 'U'")

    // 阶段二：根据阶段一的结果决定全局命运
    if err1 != nil || err2 != nil || err3 != nil {
        // 有任何失败，全局回滚，释放锁
        dbOrder.Rollback()
        dbInventory.Rollback()
        dbAccount.Rollback()
        return fmt.Errorf("事务失败，已回滚")
    }

    // 全部成功，全局提交，释放锁
    dbOrder.Commit()
    dbInventory.Commit()
    dbAccount.Commit()
    return nil
}

```

# TCC

核心思路是**业务层面的两阶段提交**。放弃数据库底层的物理长锁，改为在业务字段上做“逻辑冻结”（资源预留），从而大幅提升并发度并保证隔离性。所以说 TCC 虽然也是某种意义上的 2PA（Try，Confirm/Cancel），但是并没有采用每个服务内数据库的事务阻塞流程，而是上升到了业务逻辑去处理，从而换取了高并发场景下的吞吐量。

**执行流程：**

* **Try (预留)：**
	* 订单服务：创建订单，状态设为 `PENDING`（处理中）。
	* 库存服务：可用库存扣减 1，**冻结库存增加 1**。
	* 资金服务：可用余额扣减 100，**冻结余额增加 100**。
* **Confirm (确认)：** 如果 Try 全成功。将订单改为 `PAID`，清理库存和资金的冻结数值（真正扣除）。
* **Cancel (回滚)：** 如果 Try 失败。将订单改为 `CLOSED`，将冻结的库存和资金加回到“可用”字段中。

```go
// 伪代码：TCC 模式下的资金服务 (Account Service)
// 必须提供三个独立的 HTTP/RPC 接口

func TryDeductBalance(txID string, user string, amount int) error {
    // SQL: UPDATE acc SET balance = balance - 100, frozen = frozen + 100 WHERE user = 'U' AND balance >= 100
    // 检查并冻结资金，不真正的扣除
    return db.Execute("...")
}

func ConfirmDeductBalance(txID string, user string, amount int) error {
    // SQL: UPDATE acc SET frozen = frozen - 100 WHERE user = 'U'
    // Try 阶段已经保证了钱够了，这里只需要把冻结的钱抹掉，必须保证幂等
    return db.Execute("...")
}

func CancelDeductBalance(txID string, user string, amount int) error {
    // SQL: UPDATE acc SET balance = balance + 100, frozen = frozen - 100 WHERE user = 'U'
    // Try 失败了，把冻结的钱还给可用余额
    return db.Execute("...")
}

```

# SAGA

核心思想是乐观派，最终一致性。没有预留，没有冻结，直接把数据改掉。如果后续步骤报错，就执行反向的“补偿”操作把数据改回来。**缺点是会发生脏读**（别人能在中间状态看到你扣了钱，即使你后来回滚了）。

**执行流程：**

* **正向流转：**
1. 订单服务：直接创建 `PAID` 订单（本地事务提交）。
2. 库存服务：直接扣减 1 个库存（本地事务提交）。
3. 资金服务：尝试扣除 100 元，**发现余额不足，报错**。

* **逆向补偿：** 协调者捕获到错误，按相反顺序调用补偿接口：
1. 库存服务（补偿）：将商品 A 的库存加 1。
2. 订单服务（补偿）：将订单状态修改为 `CANCELLED`。

```go
// 伪代码：SAGA 模式下的全局编排
func PurchaseSaga() {
    saga := dtm.NewSaga("全局事务ID")
    
    // 按顺序注册 正向操作 和 对应的逆向补偿操作
    saga.Add(
        "http://order/CreateOrder",       // 正向：创建 PAID 订单
        "http://order/CancelOrder",       // 补偿：把订单改成 CANCELLED
        reqData,
    ).Add(
        "http://inventory/DeductStock",   // 正向：直接扣库存
        "http://inventory/AddStock",      // 补偿：把库存加回来
        reqData,
    ).Add(
        "http://account/DeductBalance",   // 正向：直接扣钱
        "http://account/AddBalance",      // 补偿：把钱加回来
        reqData,
    )

    // 提交给状态机引擎，由引擎负责可靠调用和失败自动回滚
    saga.Submit() 
}

```

# 可靠消息(本地消息表 + MQ)

核心思想是异步解耦，最终一致性。利用关系型数据库的 ACID 保证业务操作和“发消息”处于同一个事务中，随后由消息中间件（如 Kafka）保证消息一定能送到下游。极度适合对实时性要求不高，且系统间需要解耦的场景。

**执行流程：**

* **上游 (订单服务)：** 开启本地 MySQL 事务。向 `orders` 表插入购买记录，同时向 `local_message` 表插入两条状态为 `PENDING` 的消息（"扣库存"、"扣余额"）。提交本地事务。
* **中继 (Relay/定时任务)：** 后台进程扫描 `local_message` 表，将 `PENDING` 的消息投递到 Kafka 中，投递成功后将本地消息状态改为 `DONE`。
* **下游 (库存/资金服务)：** 监听 Kafka。收到消息后，在自己的数据库中执行扣库存/扣余额，并记录这根消息的 ID（防重表）以保证幂等性。

```go
// 伪代码：可靠消息模式下的上游订单服务
func PurchaseWithReliableMessage(user string, item string) error {
    // 开启本地数据库事务
    tx := db.Begin()
    
    // 1. 正常的业务 SQL：创建订单
    err := tx.Exec("INSERT INTO orders (user, item) VALUES (?, ?)", user, item)
    if err != nil {
        tx.Rollback()
        return err
    }

    // 2. 关键点：在同一个事务中，将要发给 Kafka 的消息写入本地表
    // 如果业务 SQL 失败，这里的消息也绝不会写入
    msgPayload1 := `{"action": "deduct_stock", "item": "A", "qty": 1}`
    msgPayload2 := `{"action": "deduct_balance", "user": "U", "amount": 100}`
    
    tx.Exec("INSERT INTO local_message (topic, payload, status) VALUES ('inventory_topic', ?, 'PENDING')", msgPayload1)
    tx.Exec("INSERT INTO local_message (topic, payload, status) VALUES ('account_topic', ?, 'PENDING')", msgPayload2)

    // 本地事务提交。只要提交成功，业务数据和消息记录一定同时落盘
    return tx.Commit()
}
```

SAGA 和可靠消息机制看起来比较相似，虽然两者都是基于本地事务串联起来的**分布式链路**，但核心分歧在于**方向与退路**：

- SAGA 允许业务失败，遇阻即触发补偿动作反向回滚。它通常伴随同步调用的延伸，适用于资金、库存等必须保证数据严格对等、且需要向用户即时反馈成败的强关联核心链路。
    
- 可靠消息没有退路（无回滚），上游落库即结束，下游依靠消息队列的无限重试直至成功。它是彻底的异步解耦，适用于发券、通知等对延迟容忍度高、且下游宕机绝不能拖累主流程的旁路业务。
