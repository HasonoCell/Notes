# MapReduce

主要就是将一次计算任务定义为 Job，分为 Map 和 Reduce 两个阶段，物理节点被分为 Master 和 Worker，Master 负责分发任务和 Worker 负责执行。

```go

// 中间结果的结构体定义
type KeyValue struct {
	Key   string
	Value string
}

// Map 函数：把输入文件拆解成键值对
// filename: 文件名 (如 pg-being_ernest.txt)
// contents: 文件的全部文本内容
func Map(filename string, contents string) []KeyValue {
	var kva []KeyValue
	// 比如遇到 "Hello World Hello"
	// 解析并产出: [{"Hello", "1"}, {"World", "1"}, {"Hello", "1"}]
	words := strings.Fields(contents) 
	for _, w := range words {
		kva = append(kva, KeyValue{w, "1"})
	}
	return kva
}

// Reduce 函数：把相同 Key 的所有 Value 聚合起来
// key: 比如 "Hello"
// values: 比如 ["1", "1"]
func Reduce(key string, values []string) string {
	// 只需要把 values 的长度加起来，就是 "Hello" 出现的总次数
	return strconv.Itoa(len(values))
}

// Worker 的主循环，Worker 设计为是 Stateless 的
func Worker(mapf func(string, string) []KeyValue, reducef func(string, []string) string) {
	for {
		// 1. 通过 RPC 向 Coordinator 申请一个 Task
		task := callAskForTask() 

		switch task.Type {
		case "MAP":
			// 拿到分配给我的文件，执行用户的 mapf
			doMapTask(task, mapf) 
			
			// 2. 关键点：将中间结果分块，写到 Worker 本地磁盘
			// writeToLocalDisk(intermediate_data)
			
			// 3. 通过 RPC 告诉 Coordinator 我干完了
			callTaskCompleted(task.ID) 

		case "REDUCE":
			// 根据 Coordinator 给的信息，通过网络去各个 Map Worker 的磁盘拉取数据
			// 执行用户的 reducef
			doReduceTask(task, reducef)
			callTaskCompleted(task.ID)

		case "WAIT":
			// 所有的 Map 任务都在 In-progress，但还没完成，
			// Reduce 阶段还没开始，Worker 先休眠一会
			time.Sleep(time.Second)

		case "EXIT":
			// 整个作业完成，Worker 退出
			return
		}
	}
}

// -----------------------------------------------------------------------

import "sync"

// 任务的内部状态
type TaskState int
const (
	Idle TaskState = iota
	InProgress
	Completed
)

// Coordinator 的核心数据结构
type Coordinator struct {
	mu          sync.Mutex      // 保护整个结构体，防止 Data Race
	mapTasks    []Task          // 记录所有 Map 任务的状态
	reduceTasks []Task          // 记录所有 Reduce 任务的状态
	
	mapDone     bool            // 所有 Map 任务是否全部 Completed？
	reduceDone  bool            // 所有 Reduce 任务是否全部 Completed？
}

// RPC Handler: Worker 调用这个方法来申请任务
func (c *Coordinator) AskForTask(args *AskTaskArgs, reply *AskTaskReply) error {
	c.mu.Lock()
	defer c.mu.Unlock() // 进门上锁，出门解锁

	if !c.mapDone {
		// 遍历 mapTasks，找出一个状态为 Idle 的任务分配给 Worker
		for i := range c.mapTasks {
			if c.mapTasks[i].State == Idle {
				c.mapTasks[i].State = InProgress
				c.mapTasks[i].StartTime = time.Now() // 记录开始时间，用于判断超时宕机
				
				// 组装 reply，把任务信息发给 Worker
				reply.Task = c.mapTasks[i]
				return nil
			}
		}
		// 如果没有 Idle 任务，但 map 还没做完（说明有人还在跑），让 Worker 先 Wait
		reply.Task = Task{Type: "WAIT"}
		return nil
	}

	// 如果 mapDone 为 true，说明可以开始派发 Reduce 任务了
	// ... 派发 Reduce 任务的逻辑（同理） ...
	
	return nil
}
```

# GFS

教材原文：

> 人们设计大型分布式系统或大型存储系统出发点通常是，他们想获取巨大的性能加成，进而利用数百台计算机的资源来同时完成大量工作。因此，性能问题就成为了最初的诉求。 之后，很自然的想法就是将数据分割放到大量的服务器上，这样就可以**并行**的从多台服务器读取数据。我们将这种方式称之为分片（Sharding）。

问题是，服务器数量一多，难免遇到服务器故障，因此需要**容错**，一种容错机制的实现思路就是**复制**：只需要维护多个数据的副本，当其中一个故障了，就可以使用另一个。但是有了多个**复制**节点就会面临改了一个节点上的数据其它节点仍然落后，也就是**一致性**的问题

GFS 解决如何分布式存储大量数据，单 Master 节点只存储元数据，它记录了哪个文件被切分成了哪些块（Chunk），以及这些块存放在哪些具体的 ChunkServer 上，它把所有元数据都放在内存里。ChunkServer 是真正存数据的地方，GFS 把文件切分成 64MB 一个的 Chunk，每个 Chunk 默认在不同的服务器上保留 3 个副本。Client（客户端）就是跑着 MapReduce 任务的机器。客户端只找 Master 获取元数据，之后客户端就直接和 ChunkServer 建立连接传输数据，**控制流和数据流是完全分离的。**

[GFS的一致性 - mit 6.824](https://night-cruise.github.io/mit-6.824/lecture-03-gfs/3.8.html) 介绍了 GFS 面临的一致性问题。


GFS 缺点在于单 Master 的性能实在是太弱了...既要面临 OOM 的风险，吞吐量也很差。我在想这也是为什么后面的 K8S 支持这种多 Master 节点（多个 api-server + etcd）的形式，从而避免**单点故障**吧。

# VMware FT

主要就是讲了如何**复制**，即通过**状态转移**和**复制状态机**两种方式来实现。

> 如果我们有一个服务器的两个副本，我们需要让它们保持同步，在实际上互为副本，这样一旦Primary出现故障，因为Backup有所有的信息，就可以接管服务。状态转移背后的思想是，Primary将自己完整状态，比如说内存中的内容，拷贝并发送给Backup。Backup会保存收到的最近一次状态，所以Backup会有所有的数据。当Primary故障了，Backup就可以从它所保存的最新状态开始运行。所以，状态转移就是发送Primary的状态。虽然VMware FT没有采用这种复制的方法，但是假设采用了的话，那么转移的状态就是Primary内存里面的内容。这种情况下，每过一会，Primary就会对自身的内存做一大份拷贝，并通过网络将其发送到Backup。为了提升效率，你可以想到每次同步只发送上次同步之后变更了的内存。
> 
> 复制状态机基于这个事实：我们想复制的大部分的服务或者计算机软件都有一些确定的内部操作，不确定的部分是外部的输入。通常情况下，如果一台计算机没有外部影响，它只是一个接一个的执行指令，每条指令执行的是计算机中内存和寄存器上确定的函数，只有当外部事件干预时，才会发生一些预期外的事。例如，某个随机时间收到了一个网络数据包，导致服务器做一些不同的事情。所以，复制状态机不会在不同的副本之间发送状态，相应的，它只会从Primary将这些外部事件，例如外部的输入，发送给Backup。通常来说，如果有两台计算机，如果它们从相同的状态开始，并且它们以相同的顺序，在相同的时间，看到了相同的输入，那么它们会一直互为副本，并且一直保持一致。**VMware 最终也采用复制状态机的模式，通过一个 LogChannel 不断将同步 Log 从 Primary 发给 Backup，实现状态的复制同步**

VMware 虽然复制了内存和 CPU 状态，但是对于磁盘数据是不会复制的，而是 Primary 和 Backup 通过网络的方式共享一个存储服务。如果主备之间的网络断了，双方都以为对方死了，都想去写共享磁盘怎么办？它们会在共享磁盘上使用原子操作（Atomic Test-and-Set）去抢夺一个锁。谁先抢到，谁就活下来变成 Primary，另一个哪怕没死也会“被自杀”。**这种机制极大地简化了选主逻辑。**

其次为了避免“Primary 刚通过网络响应外部请求，但是响应一发出去就宕机了，并且同步 Log 还没来得及发给 Backup”这种情况，VMware FT 规定了 **Output Rule**，也就是 Primary 响应外部请求之前必须收到 Backup 关于同步 Log 的 ack，确保了一致性。

既然 VMware FT 通过状态复制解决了多副本的一致性问题，为什么还需要 Raft 这种保证强一致性的分布式共识算法呢？主要就是 VMware 由于每一个中断、每一次内存非确定性读写都要同步副本，网络的带宽很快就会被 Log 填满，多核 CPU 的同步更是噩梦（论文里明确提到当时的 FT 只能支持单核 VM，多核非确定性太高，后来经过极其复杂的改进才勉强支持多核）。其次，它是“硬件级的一致性”，而我们需要“语义级的一致性”，比如我的应用只需要把 `x` 的值从 1 改成 2，VMware FT 为了同步这一步，可能要同步底层的缺页中断、寄存器入栈出栈等成百上千条 CPU 指令日志。而如果在应用层用 Raft 算法，只需要在网络上发送一条短短的日志 `"SET x=2"` 就能做到。所以 Raft 算法这种搭建在应用层的复制状态机方案，基本上性能是优于物理层的 VMware FT 这种复制状态机的。
