# 读写分离

## 什么是读写分离

单台 MySQL 数据库无论怎么优化都会遇到请求数量过多时（比如 2000 万）造成崩溃，而现实情况时，绝大多数请求都是**读多写少**（即大多数时候是在查询浏览数据，极少时候是在增删改数据），读写分离的架构因此孕育而生：

- **主库 (Master)：** 只负责**写操作**（增、删、改），处理核心数据变更，保证数据不出错。
    
- **从库 (Slave/Replica)：** 负责**读操作**，可以搞 3 台、5 台甚至 10 台从库，它们什么都不干，专门应对海量用户的查询请求。
    
- **数据同步 (Replication)：** 主库每写进去一条新数据，就会迅速通过网络复制给所有的从库，保证数据一致性。这里主从库具体的同步操作是通过 Binlog 二进制日志去实现的，主库只要发生了数据的增、删、改，都会把这些操作记录到自己的 Binlog 日志。从库通过一个专门的 I/O 线程，把新的日志内容抄回来，放到从库自己的中转站（Relay Log）里。 从库再通过一个 SQL 线程，负责把 Relay Log 里抄回来的操作，在从库自己的数据库里重新执行一遍。

## 读写分离的好处

1. **分担数据库压力：** 原本多个请求（比如 10000 个）全打在一台机器上，现在 1000 个写请求给主库，9000 个读请求平摊给 3 台从库，系统极其丝滑。
    
2. **互不干扰（避免锁竞争）：** 数据库在执行复杂写入或更新时，会把数据锁起来。如果都在一台机器上，读操作就得排队等写操作搞完。分离开后，主库写它的，从库读它的，互不阻塞。
    
3. **高可用性（容灾）：** 万一主库突然宕机，系统可以迅速把一台从库提拔为新的主库，网站不至于彻底挂掉。

# 分库分表

如果说读写分离解决的是“读多写少”的问题，那么分库分表解决的就是单台数据库连“写”操作都撑不住了的情况（比如淘宝双十一）。
