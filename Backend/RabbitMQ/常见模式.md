# RabbitMQ 常见模式笔记

这份笔记补全 RabbitMQ 四种最常见的消息模式：

- 工作队列模式
- 发布订阅模式
- 路由模式
- 主题模式

代码示例基于 Go 客户端 `github.com/rabbitmq/amqp091-go`。

## 0. 先记住一个共识

RabbitMQ 最常见的消息流转路径是：

```text
Producer -> Exchange -> Queue -> Consumer
```

但有一个例外需要记住：

- 当 `exchange` 传空字符串 `""` 时，使用的是 RabbitMQ 的默认交换机
- 这时消息会按队列名直接路由到同名队列

所以工作队列模式里经常看起来像是“直接发给队列”，其实底层仍然是经过默认交换机。

## 1. 工作队列模式

最常见的模式，适合把一批任务分发给多个 worker 处理。

典型场景：

- 订单创建后异步发短信
- 图片上传后异步生成缩略图
- 批量任务拆给多个消费者执行

### 1.1 消息流转

```text
Producer -> default exchange -> task_queue -> Consumer A
                                           -> Consumer B
                                           -> Consumer C
```

多个消费者订阅同一个队列时，RabbitMQ 默认会以轮询方式分发消息。

### 1.2 核心特点

- 一个队列
- 多个消费者竞争消费
- 一条消息通常只会被其中一个消费者处理
- 适合做任务分发和异步削峰

### 1.3 `Qos` 的作用

```go
err = ch.Qos(1, 0, false)
```

它的含义是：

- 当前消费者一次最多只保留 1 条未确认消息
- 这条消息 `Ack` 之前，不继续分更多消息给它

这样做的好处是避免慢消费者堆积过多未处理消息。

### 1.4 生产者关键代码

```go
conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
if err != nil {
	log.Fatal(err)
}
defer conn.Close()

ch, err := conn.Channel()
if err != nil {
	log.Fatal(err)
}
defer ch.Close()

q, err := ch.QueueDeclare(
	"task_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.PublishWithContext(
	context.Background(),
	"",
	q.Name,
	false,
	false,
	amqp.Publishing{
		ContentType:  "text/plain",
		DeliveryMode: amqp.Persistent,
		Body:         []byte("resize image: 1001.png"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

关键点：

- 使用默认交换机 `""`
- `routing key` 直接写队列名
- 队列和消息都建议做持久化

### 1.5 消费者关键代码

```go
conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
if err != nil {
	log.Fatal(err)
}
defer conn.Close()

ch, err := conn.Channel()
if err != nil {
	log.Fatal(err)
}
defer ch.Close()

_, err = ch.QueueDeclare(
	"task_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.Qos(1, 0, false)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(
	"task_queue",
	"",
	false,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("worker got: %s", msg.Body)
	time.Sleep(2 * time.Second)
	msg.Ack(false)
}
```

关键点：

- `autoAck` 设为 `false`
- 处理成功后再手动 `Ack`
- `ch.Qos(1, 0, false)` 很适合工作队列

### 1.6 什么时候用

当你希望“一批任务由多个 worker 分摊处理”时，用工作队列模式。

## 2. 发布订阅模式

发布订阅模式适合“一条消息要广播给多个消费者”的场景。

典型场景：

- 用户注册后同时通知邮件系统、积分系统、风控系统
- 配置变更后通知多个服务刷新缓存
- 直播开始后广播给多个下游系统

### 2.1 消息流转

```text
Producer -> fanout exchange -> queue_email    -> Consumer Email
                           -> queue_sms      -> Consumer SMS
                           -> queue_audit    -> Consumer Audit
```

### 2.2 核心特点

- 一个交换机
- 多个队列绑定到同一个交换机
- 生产者只发一次
- 每个队列都会收到一份消息

### 2.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"logs",
	"fanout",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.PublishWithContext(
	context.Background(),
	"logs",
	"",
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("user registered: 1001"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

关键点：

- `fanout` 类型会忽略 `routing key`
- 只要绑定到该交换机的队列，都会收到消息

### 2.4 消费者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"logs",
	"fanout",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

q, err := ch.QueueDeclare(
	"email_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.QueueBind(
	q.Name,
	"",
	"logs",
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(q.Name, "", true, false, false, false, nil)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("email service got: %s", msg.Body)
}
```

关键点：

- 队列通过 `QueueBind` 绑定到 `fanout exchange`
- 多个消费者通常不是消费同一个队列，而是各自消费自己的队列

### 2.5 什么时候用

当你希望“一条消息被多个系统同时收到”时，用发布订阅模式。

## 3. 路由模式

路由模式适合“同一个交换机，根据消息类别投递到不同队列”的场景。

典型场景：

- 日志系统按 `info`、`warning`、`error` 分流
- 订单事件按不同操作类型分流
- 通知系统按短信、邮件、站内信分流

### 3.1 消息流转

```text
Producer --(error)--> direct exchange --> error_queue --> Consumer Error
Producer --(info)-->  direct exchange --> info_queue  --> Consumer Info
```

### 3.2 核心特点

- 使用 `direct exchange`
- 交换机按 `routing key` 精确匹配
- 适合做分类投递

### 3.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"direct_logs",
	"direct",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

severity := "error"

err = ch.PublishWithContext(
	context.Background(),
	"direct_logs",
	severity,
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("disk almost full"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

### 3.4 消费者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"direct_logs",
	"direct",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

q, err := ch.QueueDeclare(
	"error_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.QueueBind(
	q.Name,
	"error",
	"direct_logs",
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(q.Name, "", true, false, false, false, nil)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("error consumer got: %s", msg.Body)
}
```

关键点：

- `routing key = error`
- 只有绑定 key 同样是 `error` 的队列会收到

### 3.5 什么时候用

当你希望“按明确类别精确分发消息”时，用路由模式。

## 4. 主题模式

主题模式适合“按模式匹配路由键”的场景，比路由模式更灵活。

典型场景：

- 订单事件：`order.created`、`order.paid`、`order.cancelled`
- 监控事件：`app.error.db`、`app.info.http`
- 业务领域事件总线

### 4.1 消息流转

```text
Producer --(order.created)--> topic exchange --> order_all_queue
                                              -> order_created_queue
```

### 4.2 核心特点

- 使用 `topic exchange`
- `routing key` 一般用点号分隔单词
- 支持通配符匹配

两个常见通配符：

- `*`：匹配一个单词
- `#`：匹配零个或多个单词

例如：

- `order.*` 可以匹配 `order.created`、`order.paid`
- `order.#` 可以匹配 `order.created`、`order.pay.success`

### 4.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"topic_events",
	"topic",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

eventKey := "order.created"

err = ch.PublishWithContext(
	context.Background(),
	"topic_events",
	eventKey,
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte(`{"order_id":1001}`),
	},
)
if err != nil {
	log.Fatal(err)
}
```

### 4.4 消费者关键代码

消费者 A：订阅所有订单事件

```go
q, _ := ch.QueueDeclare("order_all_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.#", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

消费者 B：只订阅创建事件

```go
q, _ := ch.QueueDeclare("order_created_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.created", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

消费者 C：订阅所有订单一级事件

```go
q, _ := ch.QueueDeclare("order_level1_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.*", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

### 4.5 什么时候用

当你希望“路由规则比 direct 更灵活，但又不想像 fanout 一样全量广播”时，用主题模式。

## 5. 四种模式怎么选

### 工作队列模式

- 一个队列
- 多个消费者竞争消费
- 一条消息通常只给一个消费者
- 适合任务分发

### 发布订阅模式

- 一个交换机绑定多个队列
- 每个队列都会收到消息
- 适合广播通知

### 路由模式

- `direct exchange`
- 按精确 `routing key` 分流
- 适合固定类别投递

### 主题模式

- `topic exchange`
- 按模式匹配 `routing key`
- 适合事件总线和灵活订阅

## 6. 一张总图

```text
1) 工作队列
Producer -> default exchange -> task_queue -> Worker A / Worker B

2) 发布订阅
Producer -> fanout exchange -> queue A -> Consumer A
                            -> queue B -> Consumer B
                            -> queue C -> Consumer C

3) 路由
Producer -(error)-> direct exchange -> error_queue
Producer -(info) -> direct exchange -> info_queue

4) 主题
Producer -(order.created)-> topic exchange -> order.created queue
                                         -> order.* queue
                                         -> order.# queue
```

## 7. 初学者最容易混淆的点

- 工作队列模式看起来像直接发队列，实际上是经过默认交换机
- 发布订阅模式里多个系统通常应该绑定不同队列，而不是共用一个队列
- 路由模式是精确匹配
- 主题模式是模式匹配
