# RabbitMQ 常见模式笔记

这份笔记补全 RabbitMQ 四种最常见的消息模式：

- 工作队列模式
- 发布订阅模式
- 路由模式
- 主题模式

代码示例基于 Go 客户端 `github.com/rabbitmq/amqp091-go`。

## 0. 先记住一个共识

RabbitMQ 最常见的消息流转路径是：

```text
Producer -> Exchange -> Queue -> Consumer
```

但有一个例外需要记住：

- 当 `exchange` 传空字符串 `""` 时，使用的是 RabbitMQ 的默认交换机
- 这时消息会按队列名直接路由到同名队列

所以工作队列模式里经常看起来像是“直接发给队列”，其实底层仍然是经过默认交换机。

## 1. 工作队列模式

最常见的模式，适合把一批任务分发给多个 worker 处理。

典型场景：

- 订单创建后异步发短信
- 图片上传后异步生成缩略图
- 批量任务拆给多个消费者执行

### 1.1 消息流转

```text
Producer -> default exchange -> task_queue -> Consumer A
                                           -> Consumer B
                                           -> Consumer C
```

多个消费者订阅同一个队列时，RabbitMQ 默认会以轮询方式分发消息。

### 1.2 核心特点

- 一个队列
- 多个消费者竞争消费
- 一条消息通常只会被其中一个消费者处理
- 适合做任务分发和异步削峰

### 1.3 `Qos` 的作用

```go
err = ch.Qos(1, 0, false)
```

它的含义是：

- 当前消费者一次最多只保留 1 条未确认消息
- 这条消息 `Ack` 之前，不继续分更多消息给它

这样做的好处是避免慢消费者堆积过多未处理消息。

### 1.4 生产者关键代码

```go
conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
if err != nil {
	log.Fatal(err)
}
defer conn.Close()

ch, err := conn.Channel()
if err != nil {
	log.Fatal(err)
}
defer ch.Close()

q, err := ch.QueueDeclare(
	"task_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.PublishWithContext(
	context.Background(),
	"",
	q.Name,
	false,
	false,
	amqp.Publishing{
		ContentType:  "text/plain",
		DeliveryMode: amqp.Persistent,
		Body:         []byte("resize image: 1001.png"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

关键点：

- 使用默认交换机 `""`
- `routing key` 直接写队列名
- 队列和消息都建议做持久化

### 1.5 消费者关键代码

```go
conn, err := amqp.Dial("amqp://guest:guest@localhost:5672/")
if err != nil {
	log.Fatal(err)
}
defer conn.Close()

ch, err := conn.Channel()
if err != nil {
	log.Fatal(err)
}
defer ch.Close()

_, err = ch.QueueDeclare(
	"task_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.Qos(1, 0, false)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(
	"task_queue",
	"",
	false,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("worker got: %s", msg.Body)
	time.Sleep(2 * time.Second)
	msg.Ack(false)
}
```

关键点：

- `autoAck` 设为 `false`
- 处理成功后再手动 `Ack`
- `ch.Qos(1, 0, false)` 很适合工作队列

### 1.6 什么时候用

当你希望“一批任务由多个 worker 分摊处理”时，用工作队列模式。

## 2. 发布订阅模式

发布订阅模式适合“一条消息要广播给多个消费者”的场景。

典型场景：

- 用户注册后同时通知邮件系统、积分系统、风控系统
- 配置变更后通知多个服务刷新缓存
- 直播开始后广播给多个下游系统

### 2.1 消息流转

```text
Producer -> fanout exchange -> queue_email    -> Consumer Email
                           -> queue_sms      -> Consumer SMS
                           -> queue_audit    -> Consumer Audit
```

### 2.2 核心特点

- 一个交换机
- 多个队列绑定到同一个交换机
- 生产者只发一次
- 每个队列都会收到一份消息

### 2.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"logs",
	"fanout",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.PublishWithContext(
	context.Background(),
	"logs",
	"",
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("user registered: 1001"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

关键点：

- `fanout` 类型会忽略 `routing key`
- 只要绑定到该交换机的队列，都会收到消息

### 2.4 消费者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"logs",
	"fanout",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

q, err := ch.QueueDeclare(
	"email_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.QueueBind(
	q.Name,
	"",
	"logs",
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(q.Name, "", true, false, false, false, nil)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("email service got: %s", msg.Body)
}
```

关键点：

- 队列通过 `QueueBind` 绑定到 `fanout exchange`
- 多个消费者通常不是消费同一个队列，而是各自消费自己的队列

### 2.5 什么时候用

当你希望“一条消息被多个系统同时收到”时，用发布订阅模式。

## 3. 路由模式

路由模式适合“同一个交换机，根据消息类别投递到不同队列”的场景。

典型场景：

- 日志系统按 `info`、`warning`、`error` 分流
- 订单事件按不同操作类型分流
- 通知系统按短信、邮件、站内信分流

### 3.1 消息流转

```text
Producer --(error)--> direct exchange --> error_queue --> Consumer Error
Producer --(info)-->  direct exchange --> info_queue  --> Consumer Info
```

### 3.2 核心特点

- 使用 `direct exchange`
- 交换机按 `routing key` 精确匹配
- 适合做分类投递

### 3.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"direct_logs",
	"direct",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

severity := "error"

err = ch.PublishWithContext(
	context.Background(),
	"direct_logs",
	severity,
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte("disk almost full"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

### 3.4 消费者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"direct_logs",
	"direct",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

q, err := ch.QueueDeclare(
	"error_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.QueueBind(
	q.Name,
	"error",
	"direct_logs",
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(q.Name, "", true, false, false, false, nil)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("error consumer got: %s", msg.Body)
}
```

关键点：

- `routing key = error`
- 只有绑定 key 同样是 `error` 的队列会收到

### 3.5 什么时候用

当你希望“按明确类别精确分发消息”时，用路由模式。

## 4. 主题模式

主题模式适合“按模式匹配路由键”的场景，比路由模式更灵活。

典型场景：

- 订单事件：`order.created`、`order.paid`、`order.cancelled`
- 监控事件：`app.error.db`、`app.info.http`
- 业务领域事件总线

### 4.1 消息流转

```text
Producer --(order.created)--> topic exchange --> order_all_queue
                                              -> order_created_queue
```

### 4.2 核心特点

- 使用 `topic exchange`
- `routing key` 一般用点号分隔单词
- 支持通配符匹配

两个常见通配符：

- `*`：匹配一个单词
- `#`：匹配零个或多个单词

例如：

- `order.*` 可以匹配 `order.created`、`order.paid`
- `order.#` 可以匹配 `order.created`、`order.pay.success`

### 4.3 生产者关键代码

```go
conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
defer conn.Close()

ch, _ := conn.Channel()
defer ch.Close()

err := ch.ExchangeDeclare(
	"topic_events",
	"topic",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

eventKey := "order.created"

err = ch.PublishWithContext(
	context.Background(),
	"topic_events",
	eventKey,
	false,
	false,
	amqp.Publishing{
		ContentType: "text/plain",
		Body:        []byte(`{"order_id":1001}`),
	},
)
if err != nil {
	log.Fatal(err)
}
```

### 4.4 消费者关键代码

消费者 A：订阅所有订单事件

```go
q, _ := ch.QueueDeclare("order_all_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.#", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

消费者 B：只订阅创建事件

```go
q, _ := ch.QueueDeclare("order_created_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.created", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

消费者 C：订阅所有订单一级事件

```go
q, _ := ch.QueueDeclare("order_level1_queue", true, false, false, false, nil)
err = ch.QueueBind(q.Name, "order.*", "topic_events", false, nil)
if err != nil {
	log.Fatal(err)
}
```

### 4.5 什么时候用

当你希望“路由规则比 direct 更灵活，但又不想像 fanout 一样全量广播”时，用主题模式。

## 5. 四种模式怎么选

### 工作队列模式

- 一个队列
- 多个消费者竞争消费
- 一条消息通常只给一个消费者
- 适合任务分发

### 发布订阅模式

- 一个交换机绑定多个队列
- 每个队列都会收到消息
- 适合广播通知

### 路由模式

- `direct exchange`
- 按精确 `routing key` 分流
- 适合固定类别投递

### 主题模式

- `topic exchange`
- 按模式匹配 `routing key`
- 适合事件总线和灵活订阅

## 6. 一张总图

```text
1) 工作队列
Producer -> default exchange -> task_queue -> Worker A / Worker B

2) 发布订阅
Producer -> fanout exchange -> queue A -> Consumer A
                            -> queue B -> Consumer B
                            -> queue C -> Consumer C

3) 路由
Producer -(error)-> direct exchange -> error_queue
Producer -(info) -> direct exchange -> info_queue

4) 主题
Producer -(order.created)-> topic exchange -> order.created queue
                                         -> order.* queue
                                         -> order.# queue
```

## 7. 死信队列

死信队列常用于处理“暂时或永久无法正常消费”的消息。

典型场景：

- 消费失败，不希望无限重试
- 消息过期后转移到另一个队列
- 队列堆满后，把溢出的消息转走
- 需要把异常消息单独保存，方便排查和人工补偿

### 7.1 先区分 `DLX` 和 `DLQ`

RabbitMQ 严格来说没有一个特殊类型叫“死信队列”。

更准确的说法是：

- `DLX`：`Dead Letter Exchange`，死信交换机
- `DLQ`：`Dead Letter Queue`，接收死信消息的普通队列

也就是说，消息并不是“直接进入死信队列”，而是：

1. 某个正常队列里的消息变成死信
2. RabbitMQ 把它重新发布到 `DLX`
3. `DLX` 再按路由规则把它投递到某个普通队列
4. 这个普通队列通常就被叫做 `DLQ`

### 7.2 什么情况下消息会变成死信

常见情况有：

- 消费者明确拒绝消息（`msg.NACK()`），并且设置 `requeue=false`
- 消息在队列中超过了 TTL
- 队列达到长度限制，旧消息被丢弃
- 对于 `quorum queue`，消息超过允许的投递次数

注意：

- “消息过期”会触发死信
- “整个队列因为过期被删除”并不等于队列里的消息会进入死信

最常见的是第一种：

```go
msg.Nack(false, false)
```

这里第二个参数是 `requeue`：

- `true`：重新入队，稍后可能再次投递
- `false`：不重新入队，如果配置了 `DLX`，消息会进入死信流程

### 7.3 工作流程

一个典型的死信流程如下：

```text
Producer
   |
   v
normal exchange
   |
   v
normal queue
   |
   | consume failed / expired / overflow
   v
dead letter exchange
   |
   v
dead letter queue
```

### 7.4 最小配置示例

下面这个例子表示：

- 正常业务队列叫 `order_queue`
- 当消息变成死信后，发到死信交换机 `order.dlx`
- 再路由到死信队列 `order_dead_queue`

先声明死信交换机和死信队列：

```go
err := ch.ExchangeDeclare(
	"order.dlx",
	"direct",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

dlq, err := ch.QueueDeclare(
	"order_dead_queue",
	true,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

err = ch.QueueBind(
	dlq.Name,
	"order.failed",
	"order.dlx",
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}
```

再声明正常业务队列，并给它加上死信参数：

```go
q, err := ch.QueueDeclare(
	"order_queue",
	true,
	false,
	false,
	false,
	amqp.Table{
		"x-dead-letter-exchange":    "order.dlx",
		"x-dead-letter-routing-key": "order.failed",
	},
)
if err != nil {
	log.Fatal(err)
}
```

这里的意思是：

- 这个队列里的消息一旦成为死信
- RabbitMQ 就把它重新发布到 `order.dlx`
- 使用 `order.failed` 作为路由键

### 7.5 消费失败后把消息送进死信队列

消费者处理失败时，常见写法如下：

```go
msgs, err := ch.Consume(
	"order_queue",
	"",
	false,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	if err := processOrder(msg.Body); err != nil {
		log.Printf("process failed, send to DLQ: %v", err)
		msg.Nack(false, false)
		continue
	}

	msg.Ack(false)
}
```

关键点：

- 处理成功就 `Ack`
- 处理失败并且不想立即重回原队列，就 `Nack(false, false)`
- 只要这个队列配置了 `DLX`，消息就会进入死信流程

### 7.6 查看死信队列中的消息

你可以再写一个专门消费死信队列的消费者，用来：

- 记录错误日志
- 告警
- 人工补偿
- 后续重试

例如：

```go
msgs, err := ch.Consume(
	"order_dead_queue",
	"",
	false,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("dead letter received: %s", msg.Body)
	log.Printf("headers: %+v", msg.Headers)
	msg.Ack(false)
}
```

RabbitMQ 会在消息头里记录死信历史，最常见的是 `x-death`，另外还会有 `x-first-death-*` 和 `x-last-death-*` 这类字段，方便你判断：

- 这条消息死信过几次
- 是因为什么原因进入死信
- 最早和最近一次是从哪个队列转出来的

### 7.7 TTL 和死信的组合

死信队列经常和 TTL 一起使用，形成“延迟重试”或“过期转移”的效果。

例如给正常队列设置 TTL：

```go
q, err := ch.QueueDeclare(
	"pay_queue",
	true,
	false,
	false,
	false,
	amqp.Table{
		"x-message-ttl":             int32(60000),
		"x-dead-letter-exchange":    "pay.dlx",
		"x-dead-letter-routing-key": "pay.timeout",
	},
)
if err != nil {
	log.Fatal(err)
}
```

含义是：

- 消息在 `pay_queue` 里最多停留 60 秒
- 超过 60 秒还没被消费，就变成死信
- 然后被投递到 `pay.dlx`

这在“订单超时未支付”“验证码过期”“任务处理超时”这类场景里很常见。

### 7.8 死信队列不等于重试队列

这是最容易误解的一点。

死信队列本质上只是“异常消息的去处”，并不自动等于“重试机制”。

如果你要做重试，通常还要额外设计：

- 重试次数
- 重试间隔
- 是否指数退避
- 最终失败后是否进入人工处理队列

也就是说：

- `DLQ` 解决的是“失败消息别直接丢”
- “自动重试”是另一层业务策略

### 7.9 生产环境里的几个建议

- 不要把所有失败消息都无脑重新入队，否则可能形成死循环
- 死信队列最好有单独的监控和告警
- 如果只是临时演示，用 `QueueDeclare` 的参数配置 `DLX` 很方便
- 如果是生产环境，RabbitMQ 官方更推荐用 policy 管理死信配置，修改起来更灵活
- 失败消息最好记录原始消息体、错误原因和业务标识，方便补偿

### 7.10 一句话总结

死信队列的本质不是“失败消息专属队列类型”，而是：

- 正常队列中的消息在某些条件下变成死信
- RabbitMQ 把它重新发布到死信交换机 `DLX`
- 再由 `DLX` 路由到一个普通队列 `DLQ`

这样异常消息不会直接消失，而是被单独保存下来，供后续排查、告警、补偿或重试。

## 8. 优先队列

优先队列用于让“更重要的消息优先被投递给消费者”。

默认情况下，RabbitMQ 队列更接近 FIFO，先进入队列的消息，通常先被投递

而优先队列会改变这个行为，优先级高的消息，可以比优先级低但更早入队的消息更先被投递

典型场景：

- 订单系统里，VIP 订单优先处理
- 消息通知里，报警消息优先于普通日志
- 任务调度里，人工紧急任务优先于批量后台任务

### 8.1 工作流程

```text
Producer --(priority=8)--> priority_queue --> Consumer
Producer --(priority=1)--> priority_queue --> Consumer
```

如果优先队列中同时存在多条不同优先级的消息，RabbitMQ 会尽量先投递优先级更高的消息。

例如：

```text
A: priority=1
B: priority=5
C: priority=3
```

它们进入同一个优先队列后，消费者更可能先收到：

```text
B -> C -> A
```

### 8.2 如何声明优先队列

优先队列不是一个单独的队列类型，而是给队列增加一个参数：

```go
"x-max-priority"
```

最小示例：

```go
q, err := ch.QueueDeclare(
	"priority_queue",
	true,
	false,
	false,
	false,
	amqp.Table{
		"x-max-priority": int32(4),
	},
)
if err != nil {
	log.Fatal(err)
}
```

这表示这个队列支持优先级，并且最大优先级设为 `4`。

几个要点：

- 没有配置 `x-max-priority` 的队列，不会按优先级排序
- 消息优先级来自发布消息时的 `Priority` 字段
- 如果消息优先级高于队列允许的最大值，会按最大值处理
- 没有设置 `Priority` 的消息，可视为优先级 `0`

### 8.3 生产者如何发送带优先级的消息

```go
err = ch.PublishWithContext(
	context.Background(),
	"",
	"priority_queue",
	false,
	false,
	amqp.Publishing{
		ContentType:  "text/plain",
		DeliveryMode: amqp.Persistent,
		Priority:     8,
		Body:         []byte("urgent task"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

再发一条低优先级消息：

```go
err = ch.PublishWithContext(
	context.Background(),
	"",
	"priority_queue",
	false,
	false,
	amqp.Publishing{
		ContentType:  "text/plain",
		DeliveryMode: amqp.Persistent,
		Priority:     1,
		Body:         []byte("normal task"),
	},
)
if err != nil {
	log.Fatal(err)
}
```

这里：

- `Priority: 8` 表示消息优先级更高
- `Priority: 1` 表示消息优先级更低

### 8.4 消费者为什么要配合 `Qos`

这是优先队列最容易误解的点。

如果消费者一连上来，就把消息立刻拿走很多条，那么消息还没来得及在队列里“排优先级”，优先队列的效果就会不明显。

因此 RabbitMQ 官方特别强调：优先队列通常要和 `basic.qos`（也就是 Go 里的 `ch.Qos(...)`）一起使用，尤其是在手动确认模式下。

示例：

```go
err = ch.Qos(1, 0, false)
if err != nil {
	log.Fatal(err)
}

msgs, err := ch.Consume(
	"priority_queue",
	"",
	false,
	false,
	false,
	false,
	nil,
)
if err != nil {
	log.Fatal(err)
}

for msg := range msgs {
	log.Printf("priority=%d body=%s", msg.Priority, msg.Body)
	time.Sleep(2 * time.Second)
	msg.Ack(false)
}
```

为什么这样更容易看到优先队列效果？

- `Qos(1, 0, false)` 限制消费者一次只拿 1 条未确认消息
- 后续消息会继续留在队列里等待调度
- RabbitMQ 才更有机会按优先级选择下一条投递给消费者

### 8.5 Classic Queue 和 Quorum Queue 的区别

根据 RabbitMQ 官方文档：

- `classic queue` 支持更常规的消息优先级能力
- `quorum queue` 只区分两档：`normal` 和 `high`

对 `quorum queue` 来说：

- 未设置优先级或优先级在 `0~4` 的消息，会视为普通优先级
- 大于 `4` 的消息，会视为高优先级

所以如果你使用的是 `quorum queue`，不要把它理解成“有很多精细优先级档位”。

### 8.6 优先队列和“多队列拆分”怎么选

官方也提醒过：很多场景下，多队列拆分比优先队列更容易理解。

例如把任务拆成：

- `task.high`
- `task.medium`
- `task.low`

很多时候会比把所有任务都塞进一个优先队列更清晰。

因为优先队列会带来这些复杂性：

- 顺序更难推断
- 和 `prefetch`、重入队、多个消费者的行为会互相影响
- 资源成本更高

所以可以这样理解：

- 任务类别很少、优先级规则简单：多队列往往更直观
- 确实需要在一个队列里做优先调度：再考虑优先队列

### 8.7 不要和“消费者优先级”混淆

RabbitMQ 还有一个概念叫 `consumer priority`，它表示：

- 多个消费者同时消费同一个队列时
- 哪个消费者更优先拿到消息

这和优先队列完全不是一回事：

- 优先队列：决定“哪条消息更先被投递”
- 消费者优先级：决定“哪个消费者更先收到消息”

一个是消息维度，一个是消费者维度。

## 9. 消息队列持久化

| 概念                         | 作用对象               | 解决什么问题                    | 常见配置/写法                                               | 不保证什么                  |
| -------------------------- | ------------------ | ------------------------- | ----------------------------------------------------- | ---------------------- |
| 队列持久化 `durable queue`      | Queue              | RabbitMQ 重启后，队列定义还在       | `QueueDeclare(..., true, ...)`                        | 不保证队列里的消息一定还在          |
| 交换机持久化 `durable exchange`  | Exchange           | RabbitMQ 重启后，交换机定义还在      | `ExchangeDeclare(..., true, ...)`                     | 不保证消息一定还在              |
| 消息持久化 `persistent message` | Message            | RabbitMQ 重启后，消息有机会恢复      | `Publishing{DeliveryMode: amqp.Persistent}`           | 不保证消息在发布瞬间绝不丢          |
| 消费者手动确认 `manual ack`       | Consumer           | 防止消费者还没处理完就把消息算成功         | `Consume(..., autoAck=false, ...)` + `msg.Ack(false)` | 不保证生产者发出的消息一定到达 Broker |
| 发布确认 `publisher confirm`   | Producer -> Broker | 让生产者知道消息是否已被 Broker 接收并确认 | `ch.Confirm(false)` + 等待 confirm                      | 不保证消费者一定处理成功           |

**把它们串起来看**

| 机制 | 更偏向哪一段链路 |
|---|---|
| `durable queue` / `durable exchange` | Broker 重启后的拓扑恢复 |
| `persistent message` | Broker 重启后的消息恢复 |
| `manual ack` | Broker -> Consumer 这段的消费可靠性 |
| `publisher confirm` | Producer -> Broker 这段的发布可靠性 |

**最常见误区**

| 误区 | 实际情况 |
|---|---|
| 开了 `durable` 就万无一失 | 不对，只有队列/交换机定义持久，不代表消息一定不丢 |
| 开了 `persistent` 就绝对安全 | 不对，消息在真正被 Broker 确认前仍可能丢 |
| 开了 `ack` 就够了 | 不对，`ack` 只解决消费端确认，不解决发布端确认 |
| 开了 `confirm` 就够了 | 不对，`confirm` 只说明 Broker 收到了，不代表消费者处理成功 |

**一个比较完整的可靠性组合**

```go
// 1. durable queue
q, err := ch.QueueDeclare("task_queue", true, false, false, false, nil)

// 2. publisher confirm
err = ch.Confirm(false)

// 3. persistent message
err = ch.PublishWithContext(ctx, "", q.Name, false, false, amqp.Publishing{
	ContentType:  "text/plain",
	DeliveryMode: amqp.Persistent,
	Body:         []byte("important task"),
})

// 4. manual ack
msgs, err := ch.Consume(q.Name, "", false, false, false, false, nil)
for msg := range msgs {
	// process...
	msg.Ack(false)
}
```

**一句话总结**

- `durable`：保拓扑
- `persistent`：保消息
- `confirm`：保发布端知道 Broker 收到没
- `ack`：保消费端处理完才算成功
