# IP 和子网掩码

IP 地址用来标识一个网络。最常用的 IPv4 地址，本质上是个 32 位二进制数字。把它每 8 位切开，换算成十进制，就成了熟悉的 `192.168.1.10` 这种格式。IP 地址分为**网络号**和**主机号**两个部分。其中，网络号就标志着这个 IP 地址所属的**子网**。在同一个子网里的 IP，通过**交换机**进行 ARP 解析得到 MAC 地址后可以直接通信；如果想跟其他子网的 IP 通信，就必须由**路由器/网关**进行转发。

问题是，给我一个 IP 地址，比如 `192.168.1`，如何分辨哪部分是网络号，哪部分是主机号？此时必须通过**子网掩码**来拆分 IP 地址。掩码也是一串 32 位的二进制数字用十进制表示，比如 `255.255.255.0`。把它换成二进制，`11111111.11111111.11111111.00000000`。子网掩码规定掩码里是 `1` 的部分，对应的 IP 就是网络号；掩码里是 `0` 的部分，对应的 IP 就是主机号。

CIDR 主要是对子网掩码进行缩写表示，比如 `192.168.1.10` 这个 IP，如果它的子网掩码是 `255.255.255.0` 即前 24 位是网络号，那不妨直接在 IP 后面加上斜线写上 1 的数量，即`192.168.1.10 /24`，这样就缩写了子网掩码。

# TUN

TUN 是 linux 内核中的虚拟网络设备。对于物理网卡（比如 eth0），一端连着真实的物理网线（负责接收光电信号），另一端连着内核的网络协议栈。而 TUN 网卡，一端连着的仍然是内核的网络协议栈，另一端连着的不是网线，而是一个**用户态的应用程序**。对内核来说，TUN 设备和真正的物理网卡没有任何区别。内核依然会给它分配 IP 地址、设置子网掩码、配置路由表，并向它发送数据包。


当在 linux 中创建了一个 TUN 设备（比如叫 `tun0`）时，操作系统会在 `/dev/net/tun` 下暴露一个 fd 给用户态的应用程序。当程序往这个文件里 `Write()` 一串字节流时，内核会认为这是从所谓的“网线”上接收到的真实数据包，然后把它推入内核的网络协议栈进行处理，最后可能从真实的物理网卡转发出去；当系统里的其他程序（比如浏览器）试图把数据发给 `tun0` 的 IP 时，内核会把这些网络包转成字节流。你的程序只需要对这个文件执行 `Read()` 操作，就能把网络包“拦截”到自己的进程内存里。

VPN 软件相信都不陌生，它就使用了 TUN 来实现代理的功能，比如：

1. VPN 软件启动，向内核申请创建了一个 `tun0` 虚拟网卡，并拿到了 fd。内核给 `tun0` 分配了 IP `10.8.0.2`。VPN 软件修改了操作系统的**路由表**，强制让所有发往国外的 IP 的数据包统统给 `tun0` 网卡。
    
2. 比如我的浏览器想访问 google。它构造了一个普通的 HTTP/IP 数据包。内核查路由表，把这个包扔进了 `tun0`。此时，包并没有发到网线上，而是掉进了 VPN 软件的程序内存里（因为 VPN 在不断 `Read` 这个文件）。
    
3. VPN 程序拿到这个原始包，对其进行**加密**，然后把它当作一段普通的业务数据，构造一个全新的 UDP 数据包里，目标地址写上海外 VPN 服务器的公网 IP。这个全新的 UDP 包通过**真实的物理网卡**发送到公网。

所以 TUN 这个名字也挺形象的，就是让用户态的程序有一个通道可以直接和内核网络栈进行沟通。针对物理网卡，字节数据直接从网线传过来；针对 TUN 虚拟网卡，既然是和用户态程序连接，字节数据又通过什么方式来呢？当然是--文件！用户态程序读写文件是非常常见且方便的事，字节数据就可以通过文件的方式写入 TUN，这是 TUN 在对用户态程序如何暴露接口这个问题上一个非常聪明的设计，用户态程序就可以接收到数据并进行处理了。此外，TUN 的本质就是一个隧道，所以 TUN 设备**不能对数据包进行任何的封装和处理**

# 网络中的常见概念

首先 TCP/UDP 是标准协议，不包括实现。一台机器如果想要支持网络通信，Linux 网络栈不可或缺，它就是用来封装数据包，再将数据包从本机通过无线网卡或者有线网卡发出去。但是 Linux 网络栈同样属于 Kernel 的一部分，按照操作系统的设计原则，用户空间程序不能直接操作 Kernel 中的网络栈，所以根据“系统调用”的概念设计了 Socket API，让用户态程序可以通过 Socket 来操作背后的网络栈，封装和发送数据包。所有用户态的程序都会通过 Socket 来实现网络通信。并且 Socket 的设计哲学和文件类似，通过 socket，close，listen，accept，connect，read 和 write 这些系统调用来操作 Socket

在微服务中，网络的拓扑结构通常极深，如果这时候直接通过底层的 Socket 或者传统的 HTTP+JSON（REST）在服务之间通信，会带来极大的开发复杂度（比如需手动处理数据序列化、粘包、连接状态）以及一定的性能损耗。

RPC 的目的就是希望服务和服务之间隐藏掉网络这一概念，调用一个远程服务就像调用自己本机上的服务一样，不用在意 URL，不用在意请求方法，不用在意网络细节，虽然 RPC 的底层依然必须基于 HTTP/TCP/UDP 进行物理层面的数据传输。

gRPC 就相当于是 RPC 的一个高性能实现框架，它不仅可以轻松方便地实现 RPC，还通过采用 HTTP/2 协议和 Protobuf 二进制压缩，真正解决了微服务海量通信时的性能损耗问题。

为什么说 RPC 性能优于 HTTP，但是底层又可以基于 HTTP 实现？这需要搞清楚 RPC 的本质，它只是用来优化开发体验，避免在调用服务的过程中不停地处理繁琐的网络细节（比如说处理状态码，序列化和反序列化传输数据等等），所以希望“能像本地函数一样调用远程函数”。REST 是面向资源的（强依赖 URL 路径和 GET/POST 等等动作），而 RPC 是面向动作/过程的。RPC 认为网络通信的细节（怎么寻址、怎么建立连接、发生网络错误怎么重试）应该被框架在底层屏蔽掉，不该污染业务代码。

我们平常用的 HTTP 1 + JSON，请求头是纯文本，不支持单个 TCP 连接的多路复用，性能较差，通常用于 Web 网页和前后端分离；而 RPC 依赖的 HTTP，通常是 HTTP 2 + Protobuf 的组合，HTTP 2 天生支持同一个 TCP 连接的多路复用，且请求头完全采用二进制格式，此外 Protobuf 比起 JSON 也有更快的序列化和反序列化速度，比起 JSON 序列化后的 ASCII 字节也有更紧凑的二进制字节数据格式。
