# IP 和子网掩码

IP 地址用来标识一个网络。最常用的 IPv4 地址，本质上是个 32 位二进制数字。把它每 8 位切开，换算成十进制，就成了熟悉的 `192.168.1.10` 这种格式。IP 地址分为**网络号**和**主机号**两个部分。其中，网络号就标志着这个 IP 地址所属的**子网**。在同一个子网里的 IP，通过**交换机**进行 ARP 解析得到 MAC 地址后可以直接通信；如果想跟其他子网的 IP 通信，就必须由**路由器/网关**进行转发。

问题是，给我一个 IP 地址，比如 `192.168.1`，如何分辨哪部分是网络号，哪部分是主机号？此时必须通过**子网掩码**来拆分 IP 地址。掩码也是一串 32 位的二进制数字用十进制表示，比如 `255.255.255.0`。把它换成二进制，`11111111.11111111.11111111.00000000`。子网掩码规定掩码里是 `1` 的部分，对应的 IP 就是网络号；掩码里是 `0` 的部分，对应的 IP 就是主机号。

CIDR 主要是对子网掩码进行缩写表示，比如 `192.168.1.10` 这个 IP，如果它的子网掩码是 `255.255.255.0` 即前 24 位是网络号，那不妨直接在 IP 后面加上斜线写上 1 的数量，即`192.168.1.10 /24`，这样就缩写了子网掩码。

# TUN

TUN 是 linux 内核中的虚拟网络设备。对于物理网卡（比如 eth0），一端连着真实的物理网线（负责接收光电信号），另一端连着内核的网络协议栈。而 TUN 网卡，一端连着的仍然是内核的网络协议栈，另一端连着的不是网线，而是一个**用户态的应用程序**。对内核来说，TUN 设备和真正的物理网卡没有任何区别。内核依然会给它分配 IP 地址、设置子网掩码、配置路由表，并向它发送数据包。


当在 linux 中创建了一个 TUN 设备（比如叫 `tun0`）时，操作系统会在 `/dev/net/tun` 下暴露一个 fd 给用户态的应用程序。当程序往这个文件里 `Write()` 一串字节流时，内核会认为这是从所谓的“网线”上接收到的真实数据包，然后把它推入内核的网络协议栈进行处理，最后可能从真实的物理网卡转发出去；当系统里的其他程序（比如浏览器）试图把数据发给 `tun0` 的 IP 时，内核会把这些网络包转成字节流。你的程序只需要对这个文件执行 `Read()` 操作，就能把网络包“拦截”到自己的进程内存里。

VPN 软件相信都不陌生，它就使用了 TUN 来实现代理的功能，比如：

1. VPN 软件启动，向内核申请创建了一个 `tun0` 虚拟网卡，并拿到了 fd。内核给 `tun0` 分配了 IP `10.8.0.2`。VPN 软件修改了操作系统的**路由表**，强制让所有发往国外的 IP 的数据包统统给 `tun0` 网卡。
    
2. 比如我的浏览器想访问 google。它构造了一个普通的 HTTP/IP 数据包。内核查路由表，把这个包扔进了 `tun0`。此时，包并没有发到网线上，而是掉进了 VPN 软件的程序内存里（因为 VPN 在不断 `Read` 这个文件）。
    
3. VPN 程序拿到这个原始包，对其进行**加密**，然后把它当作一段普通的业务数据，构造一个全新的 UDP 数据包里，目标地址写上海外 VPN 服务器的公网 IP。这个全新的 UDP 包通过**真实的物理网卡**发送到公网。

所以 TUN 这个名字也挺形象的，就是让用户态的程序有一个通道可以直接和内核网络栈进行沟通。针对物理网卡，字节数据直接从网线传过来；针对 TUN 虚拟网卡，既然是和用户态程序连接，字节数据又通过什么方式来呢？当然是--文件！用户态程序读写文件是非常常见且方便的事，字节数据就可以通过文件的方式写入 TUN，这是 TUN 在对用户态程序如何暴露接口这个问题上一个非常聪明的设计，用户态程序就可以接收到数据并进行处理了。
